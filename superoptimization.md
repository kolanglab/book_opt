# superoptimization と検証 — 最良を探し、正しさを証明する

通常のコンパイラ最適化は、人間が考えた変換を適用する。だがその変換が「本当に最良か」は誰も保証しない。
**[superoptimization](#index:superoptimization)**（スーパーオプティマイズ）は発想を逆転させる——
変換規則を捨て、「与えた仕様と**等価で、最短・最速の命令列**」を**探索で見つけ出す**[](#cite:massalin1987)。
人間のヒューリスティクスではなく、計算機の探索力で最適コードに迫る。

そして探索でコードを作るなら、当然の問いが生まれる——「見つけたコードは本当に等価か？」。
ここで**[検証](#index:検証)**（verification）が登場する。最適化の正しさを、テストではなく
**証明**で保証する技術だ[](#cite:lopes2015)。この章は「最良を探す」と「正しさを証明する」という、
最適化の二つの極北を扱う。[ピープホールの章](peephole.md)で予告した話の、本格的な回収である。

## 総当たり superoptimization — 短い順に全部試す

最初の superoptimizer は Massalin（1987）だ[](#cite:massalin1987)。アイデアは驚くほど素朴で、
だからこそ強い——**短い命令列を、短い順に全部生成し、仕様と等価かを片端から検査する**。
最初に見つかった等価な命令列が、定義上「最短」だ。

```ruby
# 概念的な総当たり superoptimizer
def superoptimize(spec, instruction_set, max_len)
  (1..max_len).each do |len|                       # 短い順に
    each_program(instruction_set, len) do |prog|   # 長さ len の命令列を全列挙
      return prog if equivalent?(prog, spec)       # 仕様と等価なら、それが最短解
    end
  end
  nil
end
```

Massalin はこれで、人間が書いたものより短い、職人技のようなビット演算列を多数発見した。
たとえば「符号関数」や「2 数の最小値」を、分岐なしの数命令で実現する技だ。
人間が思いつかない非自明な命令列が、総当たりから出てくる——これが superoptimization の衝撃だった。

問題は二つある。第一に**組み合わせ爆発**。命令列の数は長さに対して指数的に増えるので、
総当たりは短い列（数命令）にしか使えない。第二に**等価判定**だ。`equivalent?` をどう実装するか。
Massalin は多数の入力でテストして「たぶん等価」と判断したが、それでは反例を見逃しうる。
この二つの課題が、後の superoptimization 研究を駆動した。

## 等価判定を厳密にする — SMT ソルバ

「`equivalent?` をテストでなく**証明**にする」鍵が、**SMT ソルバ**（satisfiability modulo theories;
理論付き充足可能性ソルバ）だ。SMT ソルバは「この論理式を真にする値の割り当てはあるか」を
自動で判定する強力なツールで、ビットベクタや整数の理論を扱える（Z3 が有名）。

二つの命令列 P と Q が等価かを問うには、「**両者の出力が違うような入力が存在するか**」を
SMT ソルバに尋ねればよい。

```
∃ 入力. P(入力) ≠ Q(入力)     をソルバに問う
  → 「存在しない（UNSAT）」なら、すべての入力で P = Q。等価が証明された。
  → 「存在する（SAT）」なら、ソルバがその反例（違いを生む具体的な入力）を返す。
```

これは劇的な進歩だ。テストは「この入力では一致した」しか言えないが、SMT による証明は
「**すべての入力で一致する**」を保証する。しかも反例があれば具体値で示してくれるので、
バグの発見にも使える。superoptimization と検証は、この SMT という共通基盤の上で出会う。

> [!NOTE]
> SMT による等価判定にも限界はある。命令列が長い・ループを含む・メモリを複雑に触ると、
> ソルバが現実的な時間で答えを返せなくなる（SMT 自体は一般に決定不能・困難な問題を含む）。
> だから実用の superoptimizer は、短い直線コードの断片（[ピープホール](peephole.md)の窓）を
> 対象にすることが多い。[導入で述べた計算困難性](introduction.md)は、ここでも壁として立っている。

## 確率的 superoptimization — 探索を賢くする

総当たりが短い命令列にしか使えないなら、探索空間をもっと賢く歩けないか。
Schkufza、Sharma、Aiken の**確率的 superoptimization**（STOKE）[](#cite:schkufza2013)は、
全列挙をやめ、**ランダムな書き換えで命令列を少しずつ変異させながら、より良いものを探す**——
MCMC（マルコフ連鎖モンテカルロ）サンプリングを使う。

仕組みはこうだ。ある命令列から出発し、ランダムに一命令を別命令に置き換える・追加する・削除する。
変異後の命令列を、コスト関数——「**仕様との不一致度**＋**性能ペナルティ**」——で評価する。
不一致が減る・速くなる変異は受け入れやすく、悪化する変異もたまに受け入れる（局所最適から脱出するため）。
これを大量に繰り返すと、最適化された命令列に辿り着く。

```ruby
# STOKE 流の確率的探索（概念）
def stochastic_superopt(spec, init, iterations)
  current = init
  best = current
  iterations.times do
    candidate = random_mutate(current)            # 一命令をランダムに変異
    # コスト = 仕様との不一致（テスト＋反例）＋ 性能ペナルティ（遅延・サイズ）
    if accept?(cost(candidate, spec), cost(current, spec))   # Metropolis 基準
      current = candidate
      best = current if correct?(candidate, spec) && faster?(candidate, best)
    end
  end
  best   # 最後に SMT で厳密に等価検証してから採用
end
```

STOKE の妙は、「正しさ」と「速さ」を**一つのコスト関数に混ぜて**同時に最適化する点だ。
最初は間違っていてもよい——変異を重ねるうちに、まず仕様に一致し、次第に速くなる。
最終候補は SMT で厳密に検証してから採用する。確率的探索（広く速く）と SMT 検証（厳密に）を
組み合わせることで、総当たりでは届かない長さの命令列まで最適化できるようになった。
STOKE は手書きアセンブリに匹敵・凌駕するコードを生成した例を示し、注目を集めた。

> [!TIP]
> 確率的探索という発想は、本書の終盤で繰り返し現れる。[次章の機械学習による最適化](ml-and-future.md)も、
> 突き詰めれば「巨大な選択肢の空間を、勾配やサンプリングで賢く探索する」営みだ。
> 人手のヒューリスティクスの限界を、探索と学習で越える——これが最前線に共通する姿勢である。

## 最適化を検証する — Alive

superoptimization が「探索でコードを作る」なら、その双子が「人が書いた最適化規則を検証する」ことだ。
[ピープホールの章](peephole.md)で触れたように、LLVM の InstCombine は何千ものピープホール規則の集合で、
そのどれかにバグがあれば全ユーザのコードが静かに壊れる。

Lopes らの **Alive**[](#cite:lopes2015)は、ピープホール最適化規則を専用の小さな言語で
「変換前 → 変換後」の形で書くと、その正しさを**SMT ソルバで自動証明**するツールだ。
検証するのは二点——(1) すべての入力で変換前後の値が一致するか、
(2) 変換が**新たな未定義動作を導入していないか**。後者が動的言語にはない C/LLVM 特有の難所だ。

```
# Alive 風の規則記述（概念）
変換前:  %r = mul %x, 2
変換後:  %r = shl %x, 1
   → Alive が「すべての %x で mul %x,2 == shl %x,1 か？ 未定義動作は増えないか？」を SMT に問う
   → 証明できれば「正しい規則」、反例があれば具体値（例: 特定のビット幅・フラグ）を提示
```

Alive の成果は実利的だった。LLVM の既存規則を検証した結果、
**実際に複数のバグが発見・修正された**[](#cite:lopes2015)。最適化は「速くする」だけでなく
「**壊さない**」ことが至上命題だ（[導入の正当性](introduction.md)）。Alive は、人間が増やし続ける
最適化規則の正しさを、計算機が肩代わりして保証する道を開いた。

後継の **Alive2**[](#cite:lopes2021alive2)は、規則記述を書かずに **LLVM の変換そのもの**を検証する
**有界翻訳検証**（bounded translation validation）へ進んだ。最適化前後の関数を SMT で突き合わせ、
ループの展開回数などに上限を設けることで「見逃しはあっても誤警報は出さない」設計にしてある。
これを LLVM のテスト群に流すだけで、**実バグ 47 件**を発見した。「人が規則を書いて検証する」段階から、
「**コンパイラが現に行った変換を、追いかけて検証する**」段階への前進だ。
逆向きに、**Souper**[](#cite:sasnauskas2017souper)は同じ SMT 基盤を使って LLVM IR の超最適化器を作る——
検証（壊れていないか）と合成（もっと良い式はないか）が同じ土台の表裏であることを、改めて示している。

> [!IMPORTANT]
> superoptimization（探索でコードを作る）と検証（規則の正しさを証明する）は、
> SMT という同じ土台に立つ表裏一体の技術だ。前者は「もっと良いコードはないか？」を、
> 後者は「このコード変換は本当に正しいか？」を、どちらも計算機の論理推論で解く。
> 人手の規則とヒューリスティクスに頼ってきた最適化が、**機械可読な仕様と自動推論**の上に
> 再構築されつつある——これが最前線の大きな潮流である。

## まとめ

- superoptimization は変換規則を捨て、仕様と等価な最短・最速の命令列を探索で見つける[](#cite:massalin1987)。組み合わせ爆発と等価判定が課題。
- SMT ソルバは「出力が違う入力が存在するか」を問うことで、テストでなく証明として等価を判定し、反例も返す。
- 確率的 superoptimization（STOKE）[](#cite:schkufza2013)は MCMC で探索し、正しさと速さを一つのコスト関数で同時最適化する。長い命令列に届く。
- Alive[](#cite:lopes2015)はピープホール規則の正しさ（等価性＋未定義動作非導入）を SMT で自動証明し、LLVM の実バグを発見した。後継の Alive2[](#cite:lopes2021alive2)は規則でなく変換そのものを有界翻訳検証し、実バグ 47 件を発見した。

探索と論理推論で最適化を再構築する流れの先に、もう一つの大きな潮流——機械学習がある。
最終章では、ヒューリスティクスを学習で置き換える試みと、最適化基盤の未来を概観し、本書を締めくくる。
