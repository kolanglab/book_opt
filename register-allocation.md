# 有限のレジスタに値を詰める

最適化器は、好きなだけ変数（仮想レジスタ）を使ってよい IR を生む。だが本物の CPU が持つ
**物理レジスタ**は数えるほどしかない（x86-64 で汎用 16 本、うち自由に使えるのは十数本）。
無限の仮想レジスタを有限の物理レジスタに詰め込み、あふれたら主記憶へ退避する。これが
[レジスタ割り当て](#index:レジスタ割り当て)（register allocation）だ[](#cite:chaitin1982)。

レジスタ割り当ては、本書で最も「制約充足」らしい最適化だ。
レジスタアクセスはメモリアクセスより一桁速いので、いかに多くの値をレジスタに保てるかが性能を直撃する。
この章では、二大流派（グラフ彩色[](#cite:chaitin1982)[](#cite:briggs1994)と線形走査[](#cite:poletto1999)）を
Ruby で組み立て、あふれ（スピル）をいかに減らすかという核心に踏み込む。

## 干渉グラフ

出発点は単純な観察だ。二つの変数が**同じ時点で生きている**（どちらも今後使われる）なら、
同じレジスタには置けない。互いに上書きしてしまうからだ。この「同時に生きている」関係を
**[干渉](#index:干渉)**（interference）と呼び、変数を節点、干渉を辺としたグラフを
**[干渉グラフ](#index:干渉グラフ)**（interference graph）という。

干渉は[生存変数解析](dataflow-analysis.md)から作れる。各命令の時点での生存集合を求め、
「同時に生きている変数の対」すべてに辺を張る。

```ruby
# live_sets: 各プログラム点で生存している変数集合の配列
def build_interference_graph(live_sets)
  graph = Hash.new { |h, k| h[k] = Set.new }
  live_sets.each do |live|
    live.to_a.combination(2).each do |a, b|   # 同時に生きる対は干渉する
      graph[a] << b
      graph[b] << a
    end
  end
  graph
end
```

ここで割り当て問題は、グラフ理論の**[グラフ彩色](#index:グラフ彩色)**（graph coloring）に化ける。
「隣り合う節点（干渉する変数）が同じ色（レジスタ）にならないよう、K 色（K 個の物理レジスタ）で塗れるか」。
Chaitin はこの対応を見抜き、レジスタ割り当てをグラフ彩色として定式化した[](#cite:chaitin1982)。
この章で最も重要な発想の転換だ。

> [!NOTE]
> 一般のグラフを K 色で塗れるか判定するのは NP 完全だ。またしても[計算困難性](introduction.md)が現れる。
> だから実用のレジスタ割り当ては、最適解を諦めてヒューリスティクスで「たいてい良い」彩色を狙う。
> 次に見る Chaitin-Briggs 法は、その代表的なヒューリスティックだ。

## グラフ彩色によるレジスタ割り当て

Chaitin らのアルゴリズムの核は、巧妙な観察にある。
**次数が K 未満の節点は、必ず塗れる**。なぜなら、隣が K-1 個以下なら、
隣が使う色を避けてもまだ最低 1 色残るからだ。この観察を使い、**単純化**（simplify）と
**選択**（select）の二段で塗る[](#cite:chaitin1982)[](#cite:briggs1994)。

1. **単純化**: 次数 < K の節点をグラフから取り除き、スタックに積む。取り除くと残りの次数が減り、
   新たに次数 < K の節点が現れるので、それも取り除く。これを繰り返す。
2. **選択**: スタックから逆順に節点を戻し、戻すたびに「隣が使っていない色」を割り当てる。
   単純化のとき次数 < K だったので、戻す時点で必ず空き色がある。

```ruby
def color_graph(graph, k)
  stack = []
  work = graph.transform_values(&:dup)    # 破壊するので複製

  # 単純化: 次数 < k の節点を順に退避
  until work.empty?
    node = work.keys.find { |n| work[n].size < k }
    node ||= choose_spill(work)           # 見つからなければスピル候補を選ぶ（後述）
    stack.push(node)
    work[node].each { |nb| work[nb].delete(node) }   # 隣から自分への辺を消す
    work.delete(node)
  end

  # 選択: 逆順に戻して色を塗る
  color = {}
  until stack.empty?
    node = stack.pop
    used = graph[node].filter_map { |nb| color[nb] }.to_set
    avail = (0...k).find { |c| !used.include?(c) }
    if avail then color[node] = avail
    else          color[node] = :spill    # 空き色なし → 実スピル
    end
  end
  color
end
```

## スピルであふれを減らす

単純化で「次数 < K の節点が一つも見つからない」状態に陥ることがある。
グラフが K 色で塗れないかもしれないサインだ。このとき、ある変数を選んで
**[スピル](#index:スピル)**（spill）する。その値をレジスタに常駐させず主記憶に置き、
使う直前にロードし、定義直後にストアする。スピルした変数は他と干渉しなくなる（レジスタを占めないので）ので、
グラフから外して単純化を続けられる。

スピルはメモリアクセスを増やすのでできるだけ避けたい。どの変数をスピルするかが
割り当ての質を決める。良いスピル候補は「干渉が多く（多くのレジスタ圧迫を緩める）、
使用頻度が低い（スピルしてもメモリアクセスがあまり増えない）」変数だ。典型的なコスト指標は
「使用回数 ÷ 次数」で、これが小さいものを優先してスピルする。

```ruby
def choose_spill(work)
  # コスト（使用頻度）が低く、次数が高い変数ほどスピル向き
  work.keys.min_by { |n| use_count(n).to_f / work[n].size }
end
```

Chaitin の素朴版は「塗れないと分かったら即スピル」だったが、Briggs らの改良
（**楽観的彩色**）[](#cite:briggs1994)はこれを変えた。次数 ≥ K の節点もすぐスピルせず、
ひとまずスタックに積んでおく。戻すとき、隣が実際には K 色を使い切っていないことがよくあり、
そういう節点はスピルせずに塗れてしまう。「塗れないかもしれないが、楽観して後で確かめる」
というこの構えは、[SCCP](ssa-optimization.md)や[攻めの DCE](dead-code-elimination.md)で見た
「楽観から始める」精神の、レジスタ割り当て版だ。

さらに Briggs らは**コピー合体**（coalescing）も導入した。`x = y` というコピーで、
`x` と `y` が干渉しないなら、両者を同じレジスタに割り当てればコピー命令自体が消える。
これは[コピー伝播](local-optimization.md)や [SSA からの脱出](intermediate-representation.md)で
生まれたコピーを掃除する、彩色と一体の最適化だ。合体しすぎると彩色が難しくなるので、
「安全に合体できる対だけ」を見極める保守的合体が使われる。

> [!WARNING]
> スピルコードの挿入は、それ自体が新しい一時変数（ロード先）を生み、干渉グラフを変える。
> だから Chaitin 流の割り当ては「彩色 → スピル挿入 → グラフ再構築 → 再彩色」を、
> スピルがなくなるまで**繰り返す**。一回で終わらない反復プロセスである点に注意。

## 線形走査による速い割り当て

グラフ彩色は質が高いが、グラフ構築と反復彩色が重く、コンパイル時間がかかる。
実行中にコンパイルする JIT では、この遅さが致命的だ。最適化に時間をかけすぎては本末転倒になる。

そこで Poletto と Sarkar が提案したのが**[線形走査](#index:線形走査)**（linear scan）レジスタ割り当てだ
[](#cite:poletto1999)。干渉グラフを作らず、各変数の**生存区間**（live interval; 最初の定義から最後の使用までの
区間）を一本の数直線上の線分として捉える。あとはコードを頭から末尾へ一度なめながら、
「いま重なっている区間」に物理レジスタを割り当て、区間が終わったらレジスタを返すだけだ。

```ruby
# intervals: [{var:, start:, end:}] を start 順にソート済みとする
def linear_scan(intervals, num_regs)
  active = []                    # いま生きている区間（end 順）
  free = (0...num_regs).to_a     # 空きレジスタ
  alloc = {}

  intervals.each do |iv|
    # この区間の開始時点で、もう終わった区間のレジスタを解放
    active.reject! { |a| a[:end] < iv[:start] && (free << alloc[a[:var]]; true) }

    if free.empty?               # 空きなし → 最も遅く終わる区間をスピル
      victim = active.max_by { |a| a[:end] }
      if victim[:end] > iv[:end]
        alloc[iv[:var]] = alloc[victim[:var]]   # victim を追い出して横取り
        alloc[victim[:var]] = :spill
        active.delete(victim)
      else
        alloc[iv[:var]] = :spill
        next
      end
    else
      alloc[iv[:var]] = free.shift
    end
    active << iv
  end
  alloc
end
```

線形走査は、区間をソートして一度なめるだけなのでほぼ線形時間で、グラフ彩色より桁違いに速い。
質はやや劣る（生存区間を線分で近似するので、穴のある生存を粗く扱う）が、JIT には十分だ。
HotSpot のクライアントコンパイラや初期の多くの JIT がこれを採用した。
区間に「穴」を許して精度を上げる改良版（interval splitting）や、後述の SSA ベース版もある。

> [!TIP]
> グラフ彩色と線形走査の選択は、[導入で述べたコストの軸](introduction.md)の対立そのものだ。
> 「生成コードの質」を取るか「コンパイル時間」を取るか。AOT コンパイラ（GCC/LLVM の `-O2` 以上）は
> 彩色寄りの重い割り当てを、JIT は線形走査寄りの速い割り当てを選ぶ。同じ問題でも、
> いつコンパイルするか（[導入](introduction.md)）で最適なアルゴリズムが変わる。

## SSA とレジスタ割り当て

最後に近代的な視点を。長く「レジスタ割り当て＝NP 完全のグラフ彩色」と思われてきたが、
2000 年代に驚きの発見があった。**SSA 形式のプログラムの干渉グラフは弦グラフ（chordal graph）になる**[](#cite:hack2006ssa)[](#cite:hackgoos2006ipl)。
弦グラフとは、ざっくり言えば「節点をうまい順序（完全消去順序）で塗っていくと、どの節点も塗る時点で『同時に生きている隣接数 ＋ 1』色で必ず塗れる」グラフだ。
この必要色数（彩色数）はちょうど最大クリーク、すなわち最大同時生存数（レジスタ圧）に等しく、しかも多項式時間で最適に求まる。
だから[さきほどの単純化と選択](#グラフ彩色によるレジスタ割り当て)が途中で行き詰まらず、
一般グラフのように「本当は塗れるのに発見できず無駄にスピルする」ことが起きない。
ただし「弦グラフだから K 本のレジスタで必ず塗れる」わけではない。レジスタ圧が物理レジスタ数 K を超える点があれば、そこはどう塗ってもスピルが要る。
弦グラフが保証するのは「必要最小限のスピルを多項式時間で見極められる」ことであって、スピルが消えることではない。
SSA では各変数が一度しか定義されず、生存範囲が[支配木](intermediate-representation.md)に沿ってきれいに入れ子になるため、
干渉グラフが自然にこの形になるのである。

これにより「SSA のまま最適に彩色 → 必要なスピルを決める → SSA から脱出する」という、
理論的に美しく実装も見通しのよい割り当てが可能になった。とくに**彩色とスピルを分離できる**のが効く。
弦グラフは必ず塗れるので、あふれ（スピル）をどう減らすかだけを独立に考えればよい。
[SSA が局所最適化を大域化した](ssa-optimization.md)のと同じく、ここでも SSA が
「難しい問題を、構造のおかげで解きやすくする」。本書を通じた SSA の威力の、最後の現れだ。

> [!NOTE]
> **発展**：SSA ベース割り当ての実装と理論。
> スピルは、Belady の furthest-first を CFG へ一般化した手法[](#cite:braun2009spilling)などで決める。
> 別系統には、レジスタファイルをパズル盤に見立てて多項式時間で解く puzzle solving[](#cite:pereira2008puzzle)もある。
> 実装では LLVM の greedy アロケータ[](#cite:olesen2011greedy)（区間を大きい順に割り当て、衝突したら退避させて再挿入する後戻り探索）や、
> Cranelift の regalloc2[](#cite:fallin2022regalloc2)が SSA を前提に設計されている。退避やスピルの判断を
> [強化学習で置き換える試み](ml-and-future.md)[](#cite:venkatakeerthy2023rl4real)もあり、割り当て器を丸ごと検証する代わりに
> [CompCert](ml-and-future.md)が毎回**翻訳検証**する道[](#cite:rideau2010regalloc)もある。

## まとめ

- レジスタ割り当ては「同時に生きる値は同じレジスタに置けない」という干渉を、グラフ彩色として解く[](#cite:chaitin1982)。
- 次数 < K の節点は必ず塗れる、という観察で単純化と選択を行う。塗れなければスピルし、反復する。
- Briggs の楽観的彩色とコピー合体[](#cite:briggs1994)がスピルとコピーを減らす。スピル候補は「使用頻度 ÷ 次数」で選ぶ。
- 線形走査[](#cite:poletto1999)は生存区間を一度なめる高速版で JIT 向き。SSA の干渉グラフは弦グラフで、多項式時間で最適彩色できる。

レジスタを割り当てたら、最後は命令を**どの順序で並べるか**だ。
次章は、データ依存を保ったまま命令を並べ替えてパイプラインを埋める、命令スケジューリングへ進む。
