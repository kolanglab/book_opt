# 最適化とは何か

[最適化](#index:最適化)（optimization）という言葉は、数学では「最大・最小を求めること」を指すが、
言語処理系の文脈ではもう少し控えめな意味で使う。

> **最適化とは、プログラムの観測可能な振る舞い（意味）を変えずに、
> 実行コストを下げる変換のことである。**

「最適」という名前に反して、得られるのは「最適解」ではない。
後で見るように、最適なコードを得ることは一般に決定不能・計算困難であり、
実際の最適化器は「たいてい良くなる、決して悪い意味にはしない」変換の積み重ねでできている。
本書はこの控えめだが奥深い営みの地図を描く。

この章では、本書全体を貫く 4 つの問いを立てておく。
何を最適化するのか（コストの軸）、何を守るのか（正当性）、
いつ行うのか（コンパイル時か実行時か）、そしてなぜ難しいのか（理論的限界）である。

## 何を最適化するのか

「速くする」と一口に言うが、何を速くするかは一つではない。

- **実行時間**: もっとも一般的な目標。命令数を減らす、遅い命令を速い命令に置き換える、
  キャッシュミスや分岐予測ミスを減らす。
- **コードサイズ**: 組み込み機器や、命令キャッシュが小さい環境では死活問題。
  GCC/Clang の `-Os`、LLVM の `-Oz` はこの軸を最優先する。
- **メモリ使用量**: スタック・ヒープの消費。後述する[エスケープ解析](interprocedural-analysis.md)による
  ヒープ割り当ての削減などが効く。
- **消費電力 / コンパイル時間 / ウォームアップ時間**: 近年重みを増す軸。
  特に JIT では「最適化にかける時間」自体がコストに乗る。

これらの軸はしばしば対立する。ループを 8 回展開すれば実行時間は縮むかもしれないが、
コードサイズは膨れ、命令キャッシュを圧迫してかえって遅くなることもある。
だから最適化は「ある軸の改善と引き換えに、別の軸を悪化させてよいか」という
トレードオフの連続になる。`-O2` や `-O3` という最適化レベルは、
この匙加減をひとまとめにしたプリセットだと考えてよい[](#cite:aho2006)[](#cite:cooper2011)。

> [!NOTE]
> 本書は主に実行時間を題材にするが、各技法がコードサイズやコンパイル時間に
> どう響くかも随時注記する。最適化は単独では評価できず、つねに「何を犠牲にするか」とセットだ。

## 何を守るのか

最適化が**正しい**（sound）とは、変換前後でプログラムの**観測可能な振る舞い**
（observable behavior）が一致することをいう。逆に言えば、観測できない部分は自由に変えてよい。
何が「観測可能」かは言語仕様が定める。典型的には、入出力、終了の有無、
（言語が定めるなら）例外の送出や副作用の順序などである。

具体例で考えよう。次の Ruby 風コードで、

```ruby
x = a + b      # (1)
y = a + b      # (2) ← (1) と同じ計算
puts x + y
```

`a + b` を二度計算する必要はない。一度だけ計算して使い回せる（[共通部分式除去](local-optimization.md)）。

```ruby
t = a + b
x = t
y = t
puts x + y
```

この変換が正しいのは、「`a` と `b` が (1) と (2) の間で変化しない」かつ
「`a + b` の評価に観測可能な副作用がない」からだ。
もし `a` がメソッド呼び出し `a()` で、呼ぶたびにカウンタを増やすなら、
2 回呼ぶのと 1 回呼ぶのは観測上違う。この変換は不正になる。

ここに最適化の核心がある。**ほとんどの最適化は、ある前提が成り立つときだけ正しい。**
そして「その前提が本当に成り立つか」を機械的に確かめるのが、第 I 部で学ぶ
[データフロー解析](dataflow-analysis.md)の役割だ。解析は安全側に倒し、
確実でなければ「変化するかもしれない」と見なして最適化を見送る。
これを**保守的**（conservative）な解析と呼ぶ。最適化器は臆病なくらいでちょうどよい。
一度でも意味を壊せば、それはもうコンパイラのバグだからだ。

> [!WARNING]
> 「速くなるが、ごく稀に答えが変わる」最適化は、原則として採ってはならない。
> 例外は、言語仕様が明示的に許す場合（C の符号付き整数オーバーフローは未定義動作なので、
> コンパイラは「オーバーフローしない」と仮定してよい）や、`-ffast-math` のように
> ユーザが明示的に精度を放棄した場合に限られる。本書で「正しさ」と言うときは、
> つねに「言語仕様が観測可能と定める振る舞いの保存」を指す。

## いつ最適化するのか

最適化を行うタイミングは一つではない。プログラムが機械語になって走るまでの各段で、
それぞれ違う情報が手に入るからだ。

```mermaid
flowchart LR
    SRC[ソースコード] --> FE[フロントエンド<br/>字句・構文・意味解析]
    FE --> IR[中間表現 IR]
    IR --> MID[ミドルエンド<br/>機械独立な最適化]
    MID --> BE[バックエンド<br/>命令選択・割当・機械依存最適化]
    BE --> OBJ[オブジェクトコード]
    OBJ --> LTO[リンク時最適化 LTO]
    LTO --> RUN[実行]
    RUN -.プロファイル.-> JIT[実行時 JIT 最適化]
```

- **コンパイル時**（静的最適化）: ソースを見て、実行前にじっくり最適化する。
  時間をかけられるが、入力データや実行頻度は分からない。C/C++/Rust/Go の主戦場。
- **リンク時**（LTO; Link-Time Optimization）: 翻訳単位（ファイル）を越えてプログラム全体を見る。
  [手続き間最適化](whole-program-optimization.md)が一気にやりやすくなる。
- **実行時**（JIT; Just-In-Time）: 走らせながら、実際の型・分岐の偏り・ホットスポットを観測し、
  その情報を使って特殊化する。動的言語と Java/JavaScript の主戦場。
  JIT の歴史は意外に古く、1960 年代の LISP にまで遡れる[](#cite:aycock2003)。

「静的か動的か」は本書を貫く対立軸だ。静的最適化は全実行で正しい変換しかできないが、
動的最適化は「今回の実行ではこうだった」という弱い情報を、後で取り消せる**投機**として使える。
第 IV 部の[投機的最適化](speculative-optimization.md)はこの違いを最大限に活かす。

> [!NOTE]
> 本書は JIT そのものの作り方（機械語生成、コードキャッシュ管理など）には深入りしない。
> それはシリーズの『JITコンパイル入門』の領分だ。本書が扱うのは、静的・動的を問わず通用する
> **最適化のアルゴリズム**である。ただし動的でしか成立しない最適化（型特殊化など）も扱う。

## 最適化の「単位」と視野の広さ

最適化は、見渡す範囲（視野）によって段階的に分類できる。視野が広いほど大きな改善が狙えるが、
解析のコストも上がり、正しさを保証するための前提も増える。本書はおおむね狭い順に進む。

| 視野 | 名前 | 例 | 主な章 |
|------|------|-----|--------|
| 数命令 | ピープホール最適化 | `x*2` → `x<<1` | [peephole](peephole.md) |
| 基本ブロック内 | 局所最適化 | 共通部分式除去、定数畳み込み | [local-optimization](local-optimization.md) |
| 関数内（CFG 全体） | 大域最適化 | 大域定数伝播、ループ最適化 | [ssa-optimization](ssa-optimization.md), [loop-optimization](loop-optimization.md) |
| 関数をまたぐ | 手続き間最適化 | インライン展開、エスケープ解析 | [inlining](inlining.md), [interprocedural-analysis](interprocedural-analysis.md) |
| プログラム全体 | 全プログラム最適化 | 脱仮想化、LTO | [whole-program-optimization](whole-program-optimization.md) |

ここで**基本ブロック**（basic block）とは、「途中に分岐の出入りがなく、
先頭から末尾まで必ず一直線に実行される命令列」のこと。
**CFG**（control flow graph; 制御フローグラフ）とは、基本ブロックを節点、
分岐を辺とした有向グラフのこと。どちらも次章で実際に作るので、ここでは名前だけ覚えておけばよい。

## なぜ難しいのか

最後に、最適化に立ちはだかる原理的な壁を 3 つ確認しておく。
これらは「だから諦める」ための話ではなく、「だから近似とヒューリスティクスを使う」ための話だ。

**第一に、完全な最適化は決定不能である。** 「このプログラムと等価で、これ以上短くできない
プログラム」を一般に求めるアルゴリズムは存在しない。直感的には、もしそんな**万能最適化器**があれば、
それを使って「このプログラムは停止するか」まで判定できてしまう。だが停止するかどうかの判定（停止性問題）は
不可能だと知られている。だから万能最適化器も存在しない。
ゆえに「最適」を名乗る superoptimizer ですら、実際には探索範囲を限った中での最小を扱う
[](#cite:massalin1987)（第 VI 部）。

**第二に、有用な静的解析の多くは決定不能か、現実的には計算困難である。**
「この変数は実行時に必ず 0 か？」のような問いに完全に答えることはできない
（**Rice の定理**：プログラムの「意味」に関する非自明な性質は、一般に判定できない）。
だから解析は前述のとおり保守的な近似を返す。「0 かもしれない」しか言えないなら、
0 と決めつける最適化はしない。抽象解釈[](#cite:cousot1977)は、この「安全な近似」を
体系化した理論的枠組みで、[データフロー解析の章](dataflow-analysis.md)で扱う。

**第三に、フェーズ順序問題がある。** 最適化 A を先にやるべきか B を先にやるべきかで
最終結果が変わり、すべての入力に最適な固定順序は存在しない。
実際のコンパイラは経験的に決めた順序（パスパイプライン）を使い、ときに同じパスを何度も回す。
この問題に構造的に挑むのが第 VI 部の [equality saturation](equality-saturation.md) だ。

> [!IMPORTANT]
> 最適化は「正解を求める」工学ではなく、「**安全な近似を、賢いヒューリスティクスで積み重ねる**」工学である。
> 本書で各技法を読むときは、つねに次の 3 点を意識してほしい。
> (1) どんなコストを下げるのか、(2) 正しさのためにどんな前提を要求し、それを誰が保証するのか、
> (3) その前提が崩れたときどう振る舞うのか。この 3 点が、あらゆる最適化を読み解く共通の鍵になる。

次章からは、この営みの舞台となる**中間表現**を、Ruby で実際に組み立てるところから始めよう。
