# 機械学習による最適化と未来

本書を通じて、最適化器の判断の多くが**[ヒューリスティクス](#index:ヒューリスティクス)**——
経験的に調整された経験則——に支えられていることを見てきた。
[インラインすべきか](inlining.md)、[どの変数をスピルするか](register-allocation.md)、
[ループを何回展開するか](loop-optimization.md)、[どの順でパスを回すか](equality-saturation.md)。
これらは「正解の式」がなく、コンパイラ開発者がベンチマークを睨んで定数を手で回してきた領域だ。

最終章は、この「人手のヒューリスティクス」を**機械学習で置き換える**という近年の大きな潮流を扱う
[](#cite:trofin2021)。そして最適化基盤そのものの未来——MLIR[](#cite:lattner2021)のような新世代インフラ——を概観し、
本書を締めくくる。ここまで地下水脈として流れていた「人手の限界」という問題意識を、最後に地表へ引き上げる。

## なぜ最適化は機械学習と相性がよいのか

最適化の判断は、機械学習が得意とする問題の形をしている。理由を整理しよう。

- **正解がない**: 「この呼び出しをインラインすべきか」に閉じた式はない。
  あるのは「インラインした結果のプログラムが速い／小さい」という**結果**だけだ。
  これは「行動を選び、結果（報酬）で学ぶ」**強化学習**の枠組みにぴたりと合う。
- **特徴が豊富**: 呼び出し先のサイズ、ループ深さ、引数が定数か、呼び出し頻度——
  判断材料（特徴量）が数多くある。これらを入力に取る関数を学習すればよい。
- **評価が自動化できる**: 生成したコードの速さ・サイズは機械で測れる。
  学習に必要な「大量の試行と評価」を、コンパイル＆ベンチマークの自動化で回せる。

要するに、最適化のヒューリスティクスとは「特徴量から行動への写像」であり、
それはまさに機械学習が近似する対象だ。手で調整した経験則を、データから学んだモデルに置き換える——
発想は自然だが、実用コンパイラへの統合は近年まで難しかった。

## MLGO — LLVM のインライン判断を学習する

その壁を破った代表例が、LLVM の **MLGO**（Machine Learning Guided Optimizations）だ[](#cite:trofin2021)。
Trofin らは、LLVM の「**サイズ最適化のためのインライン判断**」という具体的なヒューリスティクスを、
学習したモデルに置き換えた。本物の産業用コンパイラに ML をパスとして組み込んだ、初期の本格事例だ。

仕組みはこうだ。インライン判断を「各呼び出し点で、展開する／しないを選ぶ**方策**（policy）」と捉える。
各呼び出し点の特徴量（呼び出し先サイズ、呼び出し回数、現在のコード膨張など）を入力に、
モデルが「展開する／しない」を出力する。これを**強化学習**で訓練する——
大量のプログラムをコンパイルし、最終的なコードサイズを**報酬**として、
サイズを減らす判断を強化していく。

```
従来:    特徴量 → [人手のしきい値ルール] → 展開する/しない
MLGO:    特徴量 → [学習した方策モデル]   → 展開する/しない
                         ↑
              強化学習: 多数のプログラムをコンパイルし、
              「コードサイズの削減」を報酬としてモデルを訓練
```

MLGO は、人手のヒューリスティクスを上回るコードサイズ削減を達成した[](#cite:trofin2021)。
重要なのは結果の数字だけでなく、**実用コンパイラに学習モデルを組み込む工学**が確立したことだ——
訓練したモデルを LLVM に埋め込み、コンパイル時にモデルを引いて判断する。
この基盤の上に、レジスタ割り当てや他のパスの学習も乗り始めている。

> [!NOTE]
> MLGO のような手法を実用化するには、地味だが本質的な課題がある——**再現性**と**コンパイル時間**だ。
> 学習モデルは入力にわずかな違いがあると判断が揺れうるので、同じソースが同じコードになる保証
> （決定性）を保つ工夫が要る。またモデル推論のぶんコンパイルが遅くなりすぎてはいけない。
> 「賢い判断」と「実用的な制約」の両立が、研究と製品の分かれ目になる。

## コストモデルを学習する — 性能を予測する

最適化判断には「この命令列はどれくらい速いか」というコスト予測がつきまとう。
[命令スケジューリング](instruction-scheduling.md)も[命令選択](instruction-selection.md)も、
内部に「この並びの実行サイクル数はいくつか」というコストモデルを持っている。
だが現代の CPU は、アウトオブオーダ実行・キャッシュ・分岐予測が絡み合い、
**手書きのコストモデルでは正確に予測できない**。

そこで「**コストモデル自体を学習する**」研究が進む。Ithemal（MIT）は、
基本ブロックの実行スループットを深層ニューラルネットで予測し、手書きの解析モデルや
ベンダのツールより正確な予測を達成した。命令列を入力に、実測スループットを教師として学習する——
「速さの予測」という、人間には難しくなった見積もりを、データに語らせるわけだ。

学習したコストモデルは、superoptimization やスケジューリングの**評価関数**として使える。
[前章の STOKE](superoptimization.md)が「性能ペナルティ」を必要としたように、
あらゆる探索ベースの最適化は良いコストモデルを欲しがる。
正確なコスト予測は、探索系・学習系の最適化すべての土台になる縁の下の力持ちだ。

## 強化学習でパス順序を決める

[フェーズ順序問題](equality-saturation.md)——どの最適化をどの順で回すか——も、
強化学習の格好の標的だ。「現在のプログラムの状態を見て、次に適用するパスを選ぶ」のは、
まさに「状態を見て行動を選ぶ」強化学習の問題そのものになる。

Meta の **CompilerGym** は、コンパイラ最適化を強化学習の標準的な「環境」として提供する枠組みで、
研究者が「パス順序の選択」などをゲームのように扱って学習エージェントを訓練できる。
equality saturation が「順序を選ばず全部試す」ことでフェーズ順序問題に答えたのに対し、
強化学習は「**良い順序を学習で見つける**」ことで答えようとする——同じ問題への、対照的なアプローチだ。

> [!TIP]
> ここで本書の最前線（第 VI 部）の三つの章が一本につながる。
> [equality saturation](equality-saturation.md)は「順序の問題を構造で消す」、
> [superoptimization](superoptimization.md)は「探索と論理推論で最良を見つける」、
> そして機械学習は「判断を学習で置き換える」。手段は違うが、どれも
> **「人手のヒューリスティクスの限界を、計算機の力で越える」**という同じ動機から出ている。

## 大規模言語モデルとコンパイラ

ごく最近は、大規模言語モデル（LLM）をコンパイラ最適化に使う研究も現れている。
LLVM IR を入力に「どの最適化パス列を適用すべきか」を提案させたり、コードを直接最適化させたりする
試みだ。まだ黎明期で、生成結果の**正しさをどう保証するか**が最大の課題になる——
LLM はもっともらしいが間違った変換を平気で出すので、[前章の検証](superoptimization.md)（Alive[](#cite:lopes2015)や
SMT による等価チェック）と組み合わせて「**提案は LLM、検証は論理推論**」とする方向が現実的だ。
探索・学習・検証という本書終盤の三本柱が、ここで合流する。

## 最適化基盤の未来 — MLIR

技法だけでなく、最適化を載せる**基盤**も進化している。その象徴が **MLIR**[](#cite:lattner2021)だ。
[LLVM](whole-program-optimization.md)が「一つの汎用 IR」で C 系言語を広くカバーしたのに対し、
MLIR は「**複数の IR（方言; dialect）を共存させ、段階的に下げていく**」設計を取る。

なぜこれが要るのか。深層学習のテンソル演算、GPU カーネル、量子回路、ハードウェア合成——
現代の「コンパイル対象」は多様化し、それぞれに固有の最適化（テンソル融合、
[多面体ループ変換](loop-optimization.md)など）が要る。一つの低レベル IR に早く落とすと、
これらの高レベルな最適化機会が失われてしまう。MLIR は、高レベルの方言（テンソル・アフィンループ）から
低レベルの方言（LLVM IR）へ、最適化しながら**段階的に降りる**ことで、各レベルの最適化を逃さない。

MLIR は、本書で見てきた古典的最適化（[ループ最適化](loop-optimization.md)、
[共通部分式除去](local-optimization.md)、[インライン](inlining.md)）を、
方言に依存しない再利用可能な部品として提供しつつ、領域特化の最適化をその上に積めるようにした。
「最適化を、特定の言語・IR から切り離して再利用する」——コンパイラ基盤の、次の一手だ。

## 本書を閉じるにあたって

最後に、本書を貫いた問いに戻ろう。[導入](introduction.md)で、最適化を読むときの三つの鍵を挙げた——
**何を**下げるのか、正しさのために**何を仮定し誰が保証する**のか、仮定が崩れたら**どう振る舞う**のか。

ピープホールの局所書き換えから、SSA とデータフロー解析、ループとインライン、
動的言語の投機と脱最適化、レジスタ割り当てとスケジューリング、そして equality saturation・
superoptimization・機械学習まで——技法は驚くほど多彩だったが、どれもこの三つの鍵で読み解けたはずだ。

そして全編を通じて、二つの主題が繰り返し鳴っていた。
一つは**正当性**——どんなに速くしても、意味を壊せば最適化ではない。
脱最適化[](#cite:holzle1992)も Alive[](#cite:lopes2015)も、結局は「速さと引き換えに正しさを売らない」ための装置だった。
もう一つは**計算困難性**——最適は決定不能・NP 困難で、だから私たちは安全な近似と賢いヒューリスティクスを
積み重ねる。最前線の研究（探索・学習・等価飽和）は、まさにこの「ヒューリスティクスの限界」を
越えようとする営みだった。

最適化は、完成した学問ではない。1970 年代の論文[](#cite:kildall1973)が今も現役で動き、
2020 年代の研究[](#cite:willsey2021)[](#cite:trofin2021)が日々生まれている、生きた分野だ。
本書が描いたのはその地図にすぎない。気になった技法があれば、ぜひ巻末の一次文献を開き、
自分の手で小さな最適化器を書いてみてほしい。最適化の本当の面白さは、コードが目に見えて速くなる、
あの瞬間にこそある。

## まとめ

- 最適化のヒューリスティクスは「特徴量から行動への写像」で、機械学習（特に強化学習）と相性がよい。
- MLGO[](#cite:trofin2021)は LLVM のインライン判断を学習で置き換え、人手を上回った。実用化には決定性とコンパイル時間の両立が要る。
- 学習コストモデル（Ithemal）、強化学習によるパス順序選択（CompilerGym）、LLM × 検証など、探索・学習・検証が合流しつつある。
- MLIR[](#cite:lattner2021)は方言を段階的に下げる新世代基盤で、領域特化の最適化機会を逃さず古典最適化を再利用可能にした。
- 本書を貫いたのは「正当性」と「計算困難性」。最適化は今も生きて発展する分野であり、一次文献と自作が次の一歩だ。
