# 命令スケジューリング

命令を選び、レジスタを割り当てたら、最後の仕事は並べる順序だ。
現代の CPU はパイプライン化・スーパースカラ化されていて、命令を一つずつ順番に終えるのではなく、
複数の命令を重ねて処理する。だが命令には**遅延**（latency）がある。
たとえばメモリロードは結果が出るまで数サイクルかかる。その結果をすぐ次の命令が使おうとすると、CPU は値が揃うまで
**ストール**（stall; 待ち）する。**[命令スケジューリング](#index:命令スケジューリング)**
（instruction scheduling）は、計算の意味を変えない範囲で命令を並べ替え、このストールを埋める最適化だ
[](#cite:muchnick1997)。

この章では、命令間の依存を保ちながら並べ替える基本（リストスケジューリング）から、
ループに対する強力版（ソフトウェアパイプライニング[](#cite:lam1988)）まで扱う。
鍵は、何と何の順序は守らねばならず、何は自由に動かせるかを依存グラフで切り分けることだ。

## なぜ並べ替えで速くなるのか

具体例で見よう。次の命令列で、`load` の結果が出るのに 3 サイクルかかるとする。

```
1: r1 = load [a]     ; 結果は 3 サイクル後に揃う
2: r2 = r1 + 1       ; r1 を使う → r1 がまだ来ない！ 2 サイクルのストール
3: r3 = load [b]
4: r4 = r3 + 1
```

命令 2 は命令 1 の結果 `r1` を必要とするので、`load` が終わるまで待たされる。
だが命令 3 の `load [b]` は命令 1・2 と無関係だ。これを 1 と 2 のあいだに差し込めば、
`load [a]` の待ち時間に `load [b]` の仕事をさせられる。

```
1: r1 = load [a]     ; 開始
3: r3 = load [b]     ; r1 を待つ間に b のロードを始める（待ち時間を有効活用）
2: r2 = r1 + 1       ; この頃には r1 が揃っている → ストールなし
4: r4 = r3 + 1
```

同じ 4 命令、同じ結果。だが並べ替えでストールが消えた。これがスケジューリングの利得だ。
ただし命令 2 を命令 1 より前には出せない。`r1` がまだ存在しないからだ。
動かしてよい範囲を厳密に定めるのが、次の依存グラフである。

## 依存グラフ

並べ替えで壊してはいけないのは**データ依存**だ。三種類ある[](#cite:muchnick1997)。

- **真の依存（RAW; read after write）**: 書いた値を後で読む。`r1=...` の後に `...=r1`。
  これは本質的な依存で、順序を必ず守る。
- **逆依存（WAR; write after read）**: 読んだ後にそのレジスタを書き換える。
  例：`r2 = r1 + 1`（`r1` を読む）の後に `r1 = 5`（`r1` を書く）。`r1 = 5` を前に出すと、`r2` が新しい値を読んでしまう。
- **出力依存（WAW; write after write）**: 同じレジスタに二度書く。
  例：`r1 = a` の後に `r1 = b`。順序を入れ替えると、最後に `r1` へ残る値が変わってしまう。

WAR・WAW は「同じレジスタを使い回している」ことから来る**偽の依存**で、
別のレジスタを使えば消える（上の例なら `r1 = 5` を `r3 = 5` に変えれば WAR は消え、自由に動かせる。
だから[レジスタ割り当ての後](register-allocation.md)はこれらが復活しうる）。
これらの依存を辺とした有向非巡回グラフが**[依存グラフ](#index:依存グラフ)**（dependence graph）だ。
スケジューリングは「この DAG の順序制約（辺）を守る、どんな並び（トポロジカル順序）でもよいので、
ストールが最小のものを選ぶ」問題になる。

```ruby
# instrs: 各命令の {defs:, uses:}。先行する命令との依存辺を張る。
def build_dependence_graph(instrs)
  deps = Hash.new { |h, k| h[k] = Set.new }
  instrs.each_with_index do |ins, i|
    instrs[0...i].each_with_index do |prev, j|
      raw = !(ins[:uses] & prev[:defs]).empty?   # prev が書いた値を ins が読む
      war = !(ins[:defs] & prev[:uses]).empty?   # prev が読んだ後 ins が書く
      waw = !(ins[:defs] & prev[:defs]).empty?   # 両方が同じ所に書く
      deps[i] << j if raw || war || waw          # j は i より前でなければならない
    end
  end
  deps
end
```

## リストスケジューリング

依存 DAG を守る最良の並びを求めるのは、一般には NP 困難だ（またしても）。
実用の定番は**[リストスケジューリング](#index:リストスケジューリング)**（list scheduling）という貪欲法だ
[](#cite:muchnick1997)。いま出せる命令（依存元がすべて出済み）の集合（**ready 集合**）を持ち、
そこから優先度の高い命令を選んで一つずつ出す。サイクルを進めながらこれを繰り返す。

優先度のつけ方が質を決める。よく使われるのが**クリティカルパス長**で、その命令から DAG の末端までの
最長遅延を指す。長い鎖の先頭にある命令を早く出さないと、後で詰まる。だからクリティカルパス上の命令を優先する。

```ruby
def list_schedule(instrs, deps, latency)
  scheduled = []
  done = Set.new
  cycle = 0
  ready_at = Hash.new(0)        # 各命令が出せるようになるサイクル

  until done.size == instrs.size
    ready = (0...instrs.size).select do |i|
      !done.include?(i) &&
        deps[i].all? { |d| done.include?(d) } &&   # 依存元が全部済み
        ready_at[i] <= cycle                        # 遅延も解消済み
    end
    if ready.empty?
      cycle += 1; next          # 出せる命令がない → ストール（1 サイクル進める）
    end
    pick = ready.max_by { |i| critical_path_length(i, deps, latency) }  # 優先度で選択
    scheduled << pick
    done << pick
    # この命令の結果は latency サイクル後に揃う → それを使う後続の ready_at を更新。
    # 複数依存があるので、最も遅い依存に合わせて max を取る（後の更新で早い時刻に戻さない）。
    instrs.each_index { |s| ready_at[s] = [ready_at[s], cycle + latency[pick]].max if deps[s].include?(pick) }
    cycle += 1
  end
  scheduled
end
```

リストスケジューリングは「貪欲だが、優先度を賢く選べば実用十分」という、最適化でよくある妥協の典型だ。
最適解は保証しないが、クリティカルパス優先で大半のストールを埋められる。

> [!NOTE]
> スケジューリングと[レジスタ割り当て](register-allocation.md)は互いに対立することがある。
> 並列性を増やそうと命令を散らすと、同時に生きる値が増えてレジスタ圧力が上がり、
> [スピル](register-allocation.md)が増えて逆に遅くなる。「先にスケジュールするか、先に割り当てるか」は
> 古典的なフェーズ順序問題で、両者を協調させる（同時に解く）研究も続いている。
> ここでも[導入](introduction.md)で予告した「フェーズ順序の難しさ」が顔を出す。

## トレーススケジューリング

これまでは基本ブロック 1 個の中での並べ替え（局所スケジューリング）だった。
だが基本ブロックは短く、その中だけでは埋められるストールに限りがある。
**大域スケジューリング**は、ブロックの境界を越えて命令を動かす。

代表が**トレーススケジューリング**（trace scheduling）だ。
[プロファイル](whole-program-optimization.md)でよく通る一本道（トレース）を選び、
その道に沿って分岐を無視して大きな一つのブロックのように扱い、まとめてスケジュールする。
道を外れる分岐には、辻褄を合わせる**補償コード**（compensation code）を後から挿入する。
よく通る道を速くする代わりに、めったに通らない道に後始末を押し付ける。
[投機的最適化](speculative-optimization.md)のホットな経路を優先する発想の、スケジューリング版だ。

## ソフトウェアパイプライニング

ループに対する命令スケジューリングの最強形が、Lam の**[ソフトウェアパイプライニング](#index:ソフトウェアパイプライニング)**
（software pipelining）だ[](#cite:lam1988)。アイデアは、ループの反復を**ずらして重ねる**こと。
ある反復の遅い処理（ロードなど）が終わるのを待つ間、**次の反復の早い処理**を先に始めてしまう。

```
通常のループ（各反復が直列。ロード待ちが毎回発生）
反復1: [load][      待ち      ][compute][store]
反復2:                                    [load][待ち][compute][store]

ソフトウェアパイプライン（反復をずらして重ねる）
       [load1]
       [load2][compute1]
       [load3][compute2][store1]      ← 各サイクルで複数反復の異なる段が同時進行
              [compute3][store2]
                        [store3]
```

定常状態（カーネル）では、毎サイクル「反復 i のロード・反復 i-1 の計算・反復 i-2 のストア」が
同時に走り、パイプラインが常に埋まる。立ち上がり（prologue）と締め(epilogue)の処理が要るぶん
複雑だが、ループの遅延を劇的に隠せる。Lam は **modulo scheduling** という体系的な方法でこれを求めた
[](#cite:lam1988)。VLIW プロセッサや DSP、そして自動ベクトル化と並んで、
数値計算ループの性能を支える技術だ。

[ループアンローリング](loop-optimization.md)とソフトウェアパイプライニングは、どちらも
ループの反復間に潜む並列性を引き出す点で兄弟だ。アンローリングが反復を並べて並列性を見せ、
パイプライニングが反復を重ねて並列性を作る。両者を組み合わせることも多い。

## まとめ

- 命令スケジューリングは、データ依存を保ったまま命令を並べ替え、命令遅延によるパイプラインのストールを埋める。
- 守るべきは真の依存（RAW）。偽の依存（WAR/WAW）はレジスタ使い回しから来る。依存 DAG のトポロジカル順序のうち最良を探す。
- 最適順序は NP 困難なので、リストスケジューリング（クリティカルパス優先の貪欲法）で妥協する。レジスタ割り当てとは対立しうる。
- トレーススケジューリングはホットな経路を越境スケジュールし、ソフトウェアパイプライニング[](#cite:lam1988)はループ反復を重ねて遅延を隠す。

これで第 V 部、機械寄りの最適化を一望した。最後の第 VI 部では、ここまで何度も顔を出した
二つの難問（フェーズ順序問題と局所最適）に挑む、最前線の研究へ進む。
