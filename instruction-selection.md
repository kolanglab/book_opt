# 命令選択 — 木を最小コストで覆う

ここからの第 V 部は、IR を実際の機械語に翻訳する過程**そのもの**に潜む最適化を扱う。
同じ計算でも、どの機械命令で実現するかで速度もサイズも変わる。`a*4 + b` を
「乗算＋加算」の 2 命令にするか、それとも 1 命令で済ませるか——x86 には `base + index×定数` の形を
**一発で計算する専用命令**（`lea`）があり、`a*4 + b` をこの 1 命令にまとめられる。
このように「どの命令で実現するか」を選ぶのが**[命令選択](#index:命令選択)**（instruction selection）だ[](#cite:aho2006)。

命令選択は「IR の演算を機械命令に 1 対 1 で置き換える」だけの作業に見えて、実は奥が深い。
一つの機械命令が複数の IR 演算をまとめてこなせるので、「IR の木を、命令という**部品（タイル）**で
過不足なく覆い、しかも総コストを最小にする」という最適化問題になる。この章では、
木のパターンマッチングと動的計画法で最小コストの覆いを求める方法を組み立てる。

## なぜ「選択」が最適化なのか

素朴に考えれば、`+` には `add`、`*` には `mul` を当てればよい。だが現実の CPU 命令は
もっと多芸だ。一例として、次の IR 木——配列アクセス `base[i]` のアドレス計算 `base + i*4` を見る。

```
        (+)
       /   \
    base   (*)
          /   \
         i     4
```

これを覆う方法は一通りではない。

- **素朴**: `t = i * 4`（`mul` か `shl`）、`addr = base + t`（`add`）の **2 命令**。
- **x86 の lea**: `lea addr, [base + i*4]` の **1 命令**。アドレッシングモードが
  `base + index*scale` を一発で計算する。

後者が 1 命令ぶん速く小さい。だが「`*4` と `+` をまとめて 1 命令で覆える」と気づくには、
木の**部分木のパターン**を機械命令の表と照合する必要がある。命令選択が最適化なのは、
こうした「複数の IR 演算を 1 命令に畳む機会」を、可能な覆い方の中から最良に選ぶからだ。

> [!NOTE]
> 命令選択の良し悪しは CPU のアーキテクチャに強く依る。x86 のような CISC は多芸な命令
> （複雑なアドレッシング、メモリオペランド）が多く、覆い方の選択肢が豊富だ。RISC は命令が単純で
> 選択肢が少ないぶん、後段の[スケジューリング](instruction-scheduling.md)や[レジスタ割り当て](register-allocation.md)の
> 比重が増す。「どの最適化が効くか」は機械の事情で変わる——バックエンドはとりわけ機械依存だ。

## 命令を「タイル」として捉える

命令選択の標準的な定式化が**[タイリング](#index:タイリング)**（tiling）だ。各機械命令を、
「IR の木の一部分にかぶせられる**タイル**（小さな木のパターン）」と見なす。タイルには
**コスト**（命令のサイクル数やサイズ）が付く。問題は——

> **IR の木全体を、タイルで隙間なく重なりなく覆い、コストの総和を最小にせよ。**

各タイルは「このパターンの部分木を、この 1 命令で実現できる」という規則になる。
たとえば次のようなタイル集合を考える（コストは仮の値）。

```
タイル                              生成命令              コスト
(+ reg reg)                        add                   1
(* reg const)                      imul/shl              1
(+ reg (* reg const))   ← 2 段     lea [r + r*c]         1   ← まとめ技
(load (+ reg const))               mov [r + c]           1
```

3 つ目のタイルが `base + i*4` の 2 段の部分木を 1 枚で覆う「まとめ技」だ。
このタイルを使えば 1 命令、使わなければ 2 命令。最小コストの覆いを選べば自動的に lea が選ばれる。

## 動的計画法で最小コストの覆いを解く

木をタイルで最小コストに覆う問題は、**動的計画法**（dynamic programming）できれいに解ける
[](#cite:aho2006)[](#cite:appel1998)。各ノードについて「**その部分木を覆う最小コスト**」を、
葉から根へ向かってボトムアップに計算する。あるノードを根とするタイルが何種類か当てはまるなら、
それぞれについて「タイルのコスト＋タイルが覆い残した部分木の最小コストの和」を求め、最小を選ぶ。

```ruby
# 木のノード: [op, *children]。tiles: 各タイルが「マッチ判定」と「コスト」「覆い残す子」を返す。
def min_cost_cover(node, memo = {})
  return memo[node.object_id] if memo.key?(node.object_id)

  best = Float::INFINITY
  best_tile = nil
  matching_tiles(node).each do |tile|
    # タイルが覆い残した部分木（葉として外に出る子）を再帰的に解く
    sub = tile.uncovered(node).sum { |child| min_cost_cover(child, memo) }
    total = tile.cost + sub
    if total < best
      best, best_tile = total, tile
    end
  end
  memo[node.object_id] = best
  @chosen ||= {}; @chosen[node.object_id] = best_tile   # 後で命令を出すため選択を記録
  best
end
```

最小コストを求めたら、根から `@chosen` を辿り直して、選ばれたタイルに対応する命令を出力する。
ボトムアップに最小値を求め、トップダウンに命令を吐く——典型的な DP の二段構えだ。
木の各ノードを定数回処理するので、全体は木のサイズに**線形**で動く。これは嬉しい性質だ。
命令選択は無数の覆い方を、しらみ潰しでなく DP で効率よく最適化できる。

> [!TIP]
> この「最小コストの木の覆い」を効率化・自動化した古典的な仕組みが **BURS**
> （Bottom-Up Rewrite System）や **maximal munch** だ。タイル規則の表（文法のようなもの）を
> 与えると、それを処理する命令選択器を自動生成する。LLVM の命令選択も、
> SelectionDAG / GlobalISel という形で「パターン表からマッチャを生成する」発想を受け継いでいる。
> タイル規則さえ書けば、最小コスト選択は機械的に得られる——宣言的に最適化を記述する好例だ。

## 木から DAG・グラフへ — 共有が壊す美しさ

DP による命令選択がきれいに解けるのは、対象が**木**だからだ。だが現実の IR は
[共通部分式除去](local-optimization.md)を経ると、同じ部分式が複数箇所から参照される
**DAG（有向非巡回グラフ）**になる。`(a+b) * (a+b)` の `a+b` は一つのノードを二箇所が指す。

DAG になると、木の DP は素直に使えない。あるノードを「ある覆い方で計算した結果」を
複数の親が共有するので、親ごとに別のタイルを選ぶと辻褄が合わなくなる。
実は**DAG の最適な命令選択は NP 困難**であることが知られていて、ここでも
[導入で述べた計算困難性](introduction.md)が顔を出す。実務では、DAG を木に分割してから DP を使う、
あるいはヒューリスティックなマッチングで妥協する。「木なら最適に解けるが、共有が入ると途端に難しくなる」
——最適化のあちこちで繰り返されるパターンだ。

近年は、命令選択をもっと大域的・最適に解こうとする研究も進む。SSA グラフ全体に対して、
制約ソルバ（ILP や SMT）で最適なタイリングを求めるアプローチや、
[第 VI 部の equality saturation](equality-saturation.md) で命令選択を扱う試みもある。
「局所的に貪欲なマッチ」から「大域的に最適な選択」へ——という方向は、バックエンド最適化の研究フロンティアだ。

## まとめ

- 命令選択は「IR の演算を機械命令に置く」だけでなく、「複数の IR 演算を 1 命令に畳む機会」を選ぶ最適化だ。
- 各命令をコスト付きの「タイル」と見なし、木をタイルで最小コストに覆う問題として定式化する。
- 木なら動的計画法で線形時間・最適に解ける。BURS / maximal munch はタイル規則から選択器を自動生成する。
- 共有のある DAG では最適選択は NP 困難になり、木分割やヒューリスティクスで妥協する。大域的・最適な選択は研究課題。

命令を選んだら、それらが使う値を有限の物理レジスタに割り当てねばならない。
