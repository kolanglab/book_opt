# ピープホール最適化 — 数命令を覗いて磨く

最も視野の狭い最適化から始めよう。[ピープホール最適化](#index:ピープホール最適化)
（peephole optimization）は、命令列の上を小さな「のぞき窓」（peephole）をずらしながら、
窓に入った**数命令だけ**を見て、より良い等価な命令列に置き換える手法だ。
1965 年に McKeeman が提唱した古い技法だが、今日のあらゆるコンパイラの仕上げ工程に組み込まれている[](#cite:aho2006)。

視野が狭いぶん解析もほぼ要らず、実装は素朴だ。にもかかわらず効果は侮れない。
前段の最適化やコード生成が残した「あと一歩」の無駄——冗長なロード、自明な計算、
直後にジャンプするジャンプ——を、安く確実に掃除してくれる。
この章では代表的なパターンを Ruby のルールベース変換として実装し、
最後に「人手のパターンを自動で見つける／正しさを検証する」という発展[](#cite:lopes2015)に触れる。

## のぞき窓モデル — 局所パターンの置換

ピープホール最適化の心臓は単純だ。命令列を窓（たとえば連続 2〜3 命令）で走査し、
あらかじめ用意した**書き換え規則**にマッチしたら置換する。一度置換すると
新しいマッチが生まれることがあるので、変化がなくなるまで繰り返す（不動点反復）。

代表的な書き換えパターンを分類しておく[](#cite:aho2006)[](#cite:muchnick1997)。

- **冗長ロード/ストアの除去**: `store x, R1` の直後の `load R1, x` は不要（R1 に値がある）。
- **代数的簡約と強度低減**: `x + 0 → x`、`x * 1 → x`、`x * 2 → x << 1`、`x * 0 → 0`。
- **定数畳み込み**: `2 * 3 → 6` のように、定数だけの計算をコンパイル時に済ませる。
- **制御フローの簡約**: 直後のラベルへの `goto`、ジャンプ先がまたジャンプ（ジャンプの連鎖）の短絡。
- **不要命令の除去**: `x = x`、`add R, 0` のような無効命令。

ここで**[強度低減](#index:強度低減)**（strength reduction）とは、コストの高い演算を
等価で安い演算に置き換えること。乗算をシフトや加算に、除算を乗算に——
機械にとって「同じ結果だが速い」書き換えだ。ループに対する強力版は
[ループ最適化の章](loop-optimization.md)で改めて扱う。

## ルールベースのピープホール器を書く

[前章までの三番地コード](intermediate-representation.md)を相手に、小さなピープホール器を作ろう。
ここでは窓幅 1（一命令の代数的簡約・定数畳み込み）を実装する。要は
「命令を受け取り、より良い命令か、削除指示を返す」関数の集まりだ。

```ruby
Instr = Struct.new(:op, :dst, :args)   # [intermediate-representation.md] と同じ IR

module Peephole
  module_function

  # 一命令を簡約する。書き換えなければ自身を、削除すべきなら nil を返す。
  def simplify(i)
    return i unless i.op == :binop
    op, a, b = i.args
    ca, cb = const(a), const(b)

    # 1. 定数畳み込み: 両辺が定数なら今すぐ計算
    if ca && cb
      v = ca.send(op, cb)
      return Instr.new(:assign, i.dst, [v.to_s])
    end

    # 2. 代数的簡約 / 強度低減
    case [op, ca, cb]
    in [:+, _, 0] | [:+, 0, _]      then return copy(i.dst, ca ? a : b)  # x+0 → x
    in [:*, _, 1] | [:*, 1, _]      then return copy(i.dst, cb == 1 ? a : b)
    in [:*, _, 0] | [:*, 0, _]      then return Instr.new(:assign, i.dst, ["0"])
    in [:-, _, 0]                   then return copy(i.dst, a)
    else
      # x*2 → x<<1（2 のべきの乗算をシフトに）
      if op == :* && cb && pow2?(cb)
        return Instr.new(:binop, i.dst, [:<<, a, Math.log2(cb).to_i.to_s])
      end
    end
    i
  end

  def const(s) = (Integer(s) rescue nil)             # 数字リテラルなら整数、でなければ nil
  def pow2?(n) = n > 0 && (n & (n - 1)).zero?
  def copy(dst, src) = Instr.new(:assign, dst, [src])
end
```

窓を滑らせて不動点まで回す部分はこうだ。

```ruby
def run_peephole(code)
  loop do
    changed = false
    code = code.filter_map do |i|
      ni = Peephole.simplify(i)
      changed ||= (ni != i)
      ni       # nil を返した命令は filter_map が落とす（命令削除）
    end
    break code unless changed
  end
end

src = [Instr.new(:binop, "t1", [:*, "a", "2"]),
       Instr.new(:binop, "t2", [:+, "t1", "0"]),
       Instr.new(:binop, "t3", [:*, "3", "4"])]
run_peephole(src).each { puts _1.inspect }
# t1 = a << 1   （x*2 → シフト）
# t2 = t1       （x+0 → x）
# t3 = 12       （定数畳み込み）
```

たった数十行で、乗算のシフト化・恒等元の除去・定数畳み込みが動く。
これがピープホール最適化の手触りだ。重要なのは、各規則が**局所的に等価**である限り、
全体としても等価が保たれること——窓の外を見ない安心感が、この手法の単純さを支えている。

> [!WARNING]
> 「局所的に等価」を厳密に守るのは、見た目より難しい。たとえば `x + 0 → x` は整数なら正しいが、
> **浮動小数点**では `x` が `-0.0` のとき `x + 0.0` は `+0.0` になり、結果が変わりうる。
> `x * 2 → x << 1` も、`x` が符号付きでオーバーフローしうる言語では未定義動作の扱いに注意がいる。
> ピープホール規則は「どの型・どの言語仕様で正しいか」をセットで管理しなければならない。
> この「規則の正しさ」を計算機に証明させる試みが、章末の Alive[](#cite:lopes2015)だ。

## 制御フローのピープホール — ジャンプを短絡する

データの簡約だけでなく、制御フローにも局所的な無駄が出る。
コード生成は各構文を独立に翻訳するため、つなぎ目に冗長なジャンプが残りやすい。
代表的なものを挙げる[](#cite:muchnick1997)。

- **直後ラベルへの goto**: `goto L` の直後が `L:` なら、その `goto` は消せる（自然に落ちる）。
- **ジャンプの連鎖（jump threading）**: `goto L1` で飛んだ先 `L1` がまた `goto L2` だけなら、
  最初から `goto L2` に書き換える。
- **到達不能コードの除去**: 無条件 `goto`/`ret` の直後から次のラベルまでの命令は実行されえない。
- **条件分岐の反転**: 条件が定数に畳まれたら、分岐を無条件ジャンプか直進に置換。

ジャンプの連鎖を例に実装してみる。「中身が `goto` 一個だけのブロック」を見つけ、
そこへのジャンプを最終目的地に張り替える。

```ruby
# labels: {ラベル => そのラベル位置の命令}。goto の宛先を手繰って最終目的地を返す。
def resolve_jump(label, gotos, seen = Set.new)
  return label if seen.include?(label)      # ループ防止（自己参照の goto）
  seen << label
  gotos.key?(label) ? resolve_jump(gotos[label], gotos, seen) : label
end

# gotos: {L1 => L2} は「L1 は L2 への goto 一個だけのブロック」を表す
def thread_jumps(code, gotos)
  code.map do |i|
    if i.op == :goto
      Instr.new(:goto, nil, [resolve_jump(i.args[0], gotos)])
    else
      i
    end
  end
end
```

`seen` で訪問済みラベルを覚えているのは、`L1: goto L1` のような自己ループで
無限再帰に陥らないためだ。こうした安全弁を各規則に仕込むのが、堅牢なピープホール器の作法になる。

## スーパーオプティマイザへの入口 — パターンを自動で見つける

ここまでのパターンはすべて人手で書いた。だが「ある短い命令列に対する、より速い等価な命令列」を
**人間が列挙する**のは漏れが多く、機械ごとに作り直しになる。ならば**機械に探させればよい**——
これが**[スーパーオプティマイズ](#index:スーパーオプティマイズ)**（superoptimization）の発想だ。

最初の superoptimizer は Massalin（1987）で、与えた仕様に対し、命令を短い順に
**総当たりで生成しては正しさを検査**し、最短の等価命令列を見つけ出した[](#cite:massalin1987)。
これは究極のピープホール最適化——「窓の中身を、人手の規則ではなく探索で最適化する」ものだ。
後年これは、人手のピープホール規則を自動生成する方向（peephole superoptimizer）へと発展した。

一方で、規則が増えるほど「その規則は本当に正しいのか」という不安も増す。
LLVM の InstCombine（巨大なピープホール規則の集合）の各規則を、SMT ソルバで
**自動的に検証**するツールが Alive だ[](#cite:lopes2015)。規則を専用の小言語で書くと、
「任意の入力で変換前後が等しいか、未定義動作を新たに導入していないか」を機械が証明し、
反例があれば具体値で示してくれる。実際、検証によって LLVM の既存規則からバグが見つかっている。

これらの本格的な扱いは[第 VI 部の superoptimization の章](superoptimization.md)に譲る。
ここで押さえたいのは、**最も素朴なピープホール最適化が、最も先鋭な研究（探索と検証）に
まっすぐつながっている**ことだ。「局所パターンの等価な置換」という発想は変わらず、
変わるのは「パターンを誰が・どう見つけ、正しさを誰が保証するか」だけである。

## まとめ

- ピープホール最適化は、数命令の窓を滑らせて局所パターンを等価な命令列に置換する、最も単純な最適化だ。
- 定数畳み込み・代数的簡約・強度低減・冗長ロード除去・ジャンプ短絡・到達不能コード除去が定番。
- 各規則は「どの型・どの言語仕様で局所的に等価か」を厳密に守らねばならない。浮動小数点とオーバーフローに注意。
- パターンを探索で見つける superoptimization[](#cite:massalin1987) と、規則を検証する Alive[](#cite:lopes2015) が発展の両輪。

次章では窓を基本ブロック全体に広げ、定数畳み込みを「ブロック中に伝える」定数伝播と、
冗長計算を見抜く値番号付けへ進む。
