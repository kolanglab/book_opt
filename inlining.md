# 最適化の母としてのインライン展開

ここまでの最適化はすべて「一つの関数の中」で完結していた。だが現実のプログラムは、
小さな関数やメソッドの呼び出しで編まれている。`point.x` のアクセサ、`arr.each`、
`a + b` の演算子メソッド。これらの呼び出しの境界が、関数内最適化の視野を断ち切っている。

[インライン展開](#index:インライン展開)（inlining; インライン化）は、呼び出しを呼び出し先の本体で
**置き換える**だけの単純な変換だ。だがその効果は単純ではない。呼び出しの壁が消えることで、
定数伝播も、共通部分式除去も、ループ最適化も、関数の境界を越えて働き出す。
だからインライン展開は「最適化の母」と呼ばれる。それ自体の利得より、他の最適化を呼び込む効果が本体だ
[](#cite:muchnick1997)。この章では、何が起きるのか、どこまで展開すべきか、そして展開しすぎの害を扱う。

## 呼び出しを本体で置換する

まず機械的な変換を確認しよう。呼び出し `r = f(x, y)` を、`f` の本体で置き換える。
仮引数を実引数に束縛し、`f` 内のローカル変数は名前衝突を避けてリネームし、
`f` の `return` を「結果変数への代入＋合流点へのジャンプ」に変える。

```ruby
def square(n) = n * n
def f(a)
  t = square(a + 1)    # ← この呼び出しをインライン展開
  t + 1
end

# 展開後（概念的に）
def f(a)
  _n = a + 1           # 仮引数 n に実引数を束縛
  _t = _n * _n         # square の本体（変数はリネーム）
  t = _t
  t + 1
end
```

この時点では命令はむしろ増えている。インラインの利得はここから始まる。
たとえば `f(3)` という呼び出しがあれば、[定数伝播](local-optimization.md)が `a = 3 → _n = 4 → _t = 16`
と伝わり、`f(3)` 全体が `17` に畳まれる。展開しなければ、`square` の中身は `f` から見えず、
この畳み込みは決して起きなかった。

直接の利得もある。呼び出し規約のオーバーヘッド（引数をレジスタやスタックに積む、
呼び出し命令、フレームの設定と破棄、戻り値の受け渡し）が丸ごと消える。
本体が数命令しかないアクセサ `point.x` では、この規約コストが本体より大きいことすらある。
だから**小さな関数ほどインラインの費用対効果が高い**。

> [!NOTE]
> 動的言語では、そもそも「`a + b` の `+` がどの関数か」が実行時まで分からないため、
> 静的にインラインできない。だが実行時に「今回は `Integer#+` だった」と観測すれば、
> その関数を投機的にインラインできる。これが第 IV 部の[投機的最適化](speculative-optimization.md)の
> 中心的な利得だ。静的言語のインラインと動的言語の投機的インラインは、地続きの技術である。

## インラインヒューリスティクス

すべての呼び出しを展開できれば話は早いが、そうはいかない。展開はコードを増やし、
再帰呼び出しは無限に展開しうる。だから「展開すべきか」を判断する**[ヒューリスティクス](#index:ヒューリスティクス)**
（heuristic; 経験則）が要る。実際のコンパイラが見る要素は概ねこうだ[](#cite:muchnick1997)。

- **呼び出し先の大きさ**: 小さい関数は展開（規約コストが相対的に大きい）。
- **呼び出し頻度**: ホットな呼び出し点ほど展開する価値が高い（**ホット**＝実行時に何度も通る部分、**コールド**＝めったに通らない部分。ループ内やプロファイルで頻出ならホット）。
- **引数が定数か**: 定数引数があると、展開後に大きく畳み込める見込み。
- **再帰の深さ**: 再帰は有限回（または末尾なら別扱い）に制限する。
- **総コード膨張の予算**: 関数全体・プログラム全体での膨張に上限を設ける。

素朴な実装は「本体サイズ ≤ しきい値、かつ展開後の予算内」なら展開、というものだ。

```ruby
# call_site: {callee:, in_loop:, const_args:}, budget: 残りコード膨張予算
def should_inline?(call_site, callee_size, budget)
  return false if callee_size > budget
  score = 0
  score += 30 if callee_size <= 5          # ごく小さい関数は強く推す
  score += 20 if call_site[:in_loop]       # ホットなら加点
  score += 15 if call_site[:const_args]    # 定数畳み込みが見込める
  score -= callee_size                     # 大きいほど減点
  score > 0
end
```

このしきい値ベースの調整は、コンパイラ開発で最も「黒魔術」と呼ばれてきた部分だ。
ベンチマークを睨んで定数を手で回す。その属人性ゆえに、近年は**インライン判断を機械学習で
置き換える**研究が進んでいる。LLVM の MLGO[](#cite:trofin2021) は、「サイズ最適化のための
インライン判断」を強化学習したモデルに置き換え、人手のヒューリスティクスを上回るコード削減を示した。
[第 VI 部](ml-and-future.md)で改めて扱う。

## 再帰と末尾呼び出し

再帰関数はそのままでは無限展開になる。実務では「深さ k まで展開して、残りは普通の呼び出しに戻す」
のが基本だ。だが特別扱いできる重要なケースがある。**[末尾呼び出し](#index:末尾呼び出し)**
（tail call）だ。

関数の最後の動作が別の関数（自分自身を含む）の呼び出しで、その戻り値をそのまま返すとき、
呼び出し元のスタックフレームはもう使われない。だから新しいフレームを積まず、現在のフレームを
再利用できる。これが**末尾呼び出し最適化**（tail call optimization; TCO）で、
末尾再帰の場合は実質ループに変換される。

```ruby
# 末尾再帰の階乗。最後の動作が fact_iter の呼び出し（戻り値をそのまま返す）
def fact_iter(n, acc)
  return acc if n == 0
  fact_iter(n - 1, acc * n)   # ← 末尾呼び出し。フレームを再利用すればループになる
end

# TCO 後（概念的に）: スタックを消費しないループ
def fact_iter(n, acc)
  loop do
    return acc if n == 0
    n, acc = n - 1, acc * n   # フレーム再利用 = 引数の更新
  end
end
```

TCO はスタックオーバーフローを防ぎ、関数型スタイルのコードを実用的にする。
Scheme は言語仕様で TCO を義務づけている。一方 Ruby や Python は、デバッグ時の
スタックトレースを優先して標準では行わない（Ruby は `compile_option` で有効化可能）。
「最適化が観測可能な振る舞い（ここではスタックトレース）を変えてよいか」という、
[導入で述べた正当性](introduction.md)の線引きが、言語ごとの設計判断として現れる好例だ。

## 部分インライン

大きな関数は丸ごと展開すると膨れるが、よく見ると入口の早期 return だけがホットなことがある。

```ruby
def lookup(key)
  return @cache[key] if @cache.key?(key)   # ほとんどここで返る（ホット・小さい）
  slow_compute(key)                        # まれにしか来ない（コールド・大きい）
end
```

この `lookup` をそのままインラインすると `slow_compute` まで連れてくるが、
ホットな前半だけを展開し、コールドな後半は元の関数呼び出しとして残せば、
膨張を抑えつつ利得を取れる。これが**部分インライン**（partial inlining）だ。
関数をホットな入口部分とコールドな残りに割り、入口だけ展開する。
プロファイル情報があると、こうした賢い切り分けができる。

> [!TIP]
> インラインは「呼び出しを消す」だけの変換に見えて、実は**プログラム全体のコード配置に効く**。
> ホットなコードを集めてコールドなコードを追い出すと、命令キャッシュの利用効率が上がる。
> 部分インラインや、後述の[全プログラム最適化](whole-program-optimization.md)での
> ホット/コールド分離は、この「局所性」のための最適化でもある。

## 膨張とフェーズ順序という害

インラインは万能ではない。害も正しく見ておこう。

第一に**コード膨張**。展開しすぎると命令キャッシュを溢れさせ、かえって遅くなる。
特に「小さい関数を、多くの呼び出し点で展開する」と、同じ本体のコピーが大量にできる。

第二にコンパイル時間。展開で関数が巨大になると、その後の最適化（とくに
レジスタ割り当てやデータフロー解析）のコストが関数サイズに対して超線形に効く
（サイズが 2 倍になると手間は 2 倍では済まず、それ以上に増える）。

第三に、最も厄介な**[フェーズ順序問題](introduction.md)**。インラインは他の最適化を呼び込むが、
「いつインラインすべきか」は他の最適化の結果に依存する。早くインラインしすぎると、
まだ畳まれていない大きな本体を展開して膨らみ、遅すぎると最適化機会を逃す。
さらに「A を展開すると B の展開予算が減る」という相互作用もある。
インラインの順序と判断は、コンパイラ最適化で最も難しい大域的決定の一つだ。

これらの害ゆえに、インライン判断には正確な情報が欲しくなる。
この呼び出しは本当にホットか、展開後にどれだけ畳めるか、この関数は副作用を持つか。
その情報を供給するのが、次章の[手続き間解析](interprocedural-analysis.md)だ。

## まとめ

- インライン展開は呼び出しを本体で置換する単純な変換だが、関数の壁を壊して他の全最適化を呼び込む「最適化の母」。
- 展開すべきかはヒューリスティクス（サイズ・頻度・定数引数・予算）で決め、その調整は近年 ML 化されつつある[](#cite:trofin2021)。
- 末尾呼び出しはフレームを再利用してループ化でき、再帰を実用にする。言語ごとに方針が分かれる。
- 部分インラインはホットな入口だけ展開して膨張を抑える。インラインの害は膨張・コンパイル時間・フェーズ順序。

次章では、インラインを賢く行うにも大きな最適化を行うにも必要な「呼び先・別名・エスケープ」を
解析する、手続き間解析の三種の神器を学ぶ。
