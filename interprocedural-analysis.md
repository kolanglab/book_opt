# 手続き間解析

関数の境界を越える最適化には、境界を越える解析が要る。
「この呼び出しはどの関数に行き着くか」（コールグラフ）、
「このポインタはどのオブジェクトを指すか」（別名解析）、
「このオブジェクトは関数の外に漏れるか」（エスケープ解析）。
この三つが分かって初めて、インラインや脱仮想化、スタック割り当てといった大きな最適化が安全に行える。

[手続き間解析](#index:手続き間解析)（interprocedural analysis）は、関数内（イントラプロシージャル）解析より
ずっと難しい。再帰や関数ポインタ、動的ディスパッチが、解析の対象であるグラフ自体を不確かにするからだ。
この章では精度とコストのトレードオフを軸に、三種の神器を Ruby で組み立てる。

## コールグラフ

すべての出発点は**[コールグラフ](#index:コールグラフ)**（call graph）だ。
関数を節点、「f が g を呼ぶ」を辺とする有向グラフで、これがないと「プログラム全体」を辿れない。

静的な直接呼び出し（`g()` と名指し）だけなら、コールグラフは構文から自明に作れる。
難しいのは**間接呼び出し**、つまり関数ポインタ、高階関数、そして動的ディスパッチ（`obj.m()`）だ。
`obj.m()` がどの `m` を呼ぶかは `obj` の型による。だが `obj` の型が静的に一意でないと、
「呼びうる関数すべて」への辺を張るしかない。コールグラフの精度は、この間接呼び出しの絞り込み精度で決まる。

```ruby
# 直接呼び出しからコールグラフを作る素朴版
def build_call_graph(functions)
  graph = Hash.new { |h, k| h[k] = Set.new }
  functions.each do |name, body|
    body.each do |instr|
      graph[name] << instr.args[0] if instr.op == :call   # 直接呼び出しの宛先
    end
  end
  graph
end
```

オブジェクト指向言語で `obj.m()` の呼び先を絞る手法は段階的だ。
最も粗いのは「`m` という名前のメソッドすべて」。一段精密なのが**クラス階層解析（CHA）**で、
`obj` の静的型のサブクラスにある `m` だけに限る[](#cite:dean1995)。さらに**Rapid Type Analysis（RTA）**は、
「プログラム中で実際に `new` されるクラス」だけに絞る[](#cite:bacon1996)。これらは
[全プログラム最適化の章](whole-program-optimization.md)で脱仮想化として詳しく扱う。
ここでは「コールグラフは間接呼び出しの解決精度に左右される、最適化の地図だ」と押さえておく。

## 別名解析

関数間で最も悩ましいのが**[別名](#index:別名)**（alias）だ。二つのポインタ／参照 `p` と `q` が
同じオブジェクトを指しうるとき、それらは別名（alias）であるという。
別名が分からないと、`*p = 1` の後に `*q` を読んでも値が変わったか分からず、
[共通部分式除去](local-optimization.md)も[定数伝播](ssa-optimization.md)も、メモリをまたぐと一切できなくなる。

```ruby
a = SomeObject.new
b = a              # b は a と同じオブジェクトを指す（別名！）
a.x = 1
b.x = 2            # これは a.x も変える
puts a.x           # 2。もし「a.x=1 だから 1」と最適化したら間違い
```

別名を解く**[指示先解析](#index:指示先解析)**（points-to analysis; ポインタ解析）には、
精度とコストの異なる代表的な二流派がある。

**Andersen 流（包含ベース）**: `p = q` を「q の指す先 ⊆ p の指す先」という包含制約として集め、
制約を解く。精密だが、最悪 O(n³) と重い。

**Steensgaard 流（統一ベース）**: `p = q` を「p と q は同じ点集合」と**統一**してしまう
[](#cite:steensgaard1996)。精度は落ちる（一方向の代入も双方向に見なす）が、
**Union-Find**（「同じグループか」の判定と二つのグループの併合を、ほぼ定数時間でこなすデータ構造）で
ほぼ線形時間に解け、巨大プログラムでも回る。Steensgaard 自身が
「ほぼ線形時間で」と題したとおり、スケーラビリティが売りだ。

```ruby
# Steensgaard 流の核: 代入で両辺の「指す先」を統一する（Union-Find）
class Steensgaard
  def initialize = @parent = {}
  def find(x) = @parent[x] == x || @parent[x].nil? ? (@parent[x] = x) : (@parent[x] = find(@parent[x]))
  def unify(a, b)                      # p = q を見たら呼ぶ
    ra, rb = find(a), find(b)
    @parent[ra] = rb unless ra == rb   # 同じ点集合にまとめる
  end
  def alias?(a, b) = find(a) == find(b)
end
```

二流派の差を一例で見よう。`a = &x; b = &y; b = a` というコードで、Andersen は片方向に追って
「`a` は x を、`b` は x と y を指す」と分かる。だが Steensgaard は `b = a` を見た瞬間に `a` と `b` を
同じ集合へ統一するので、「`a` も `b` も x と y を指す」と、本来 x しか指さない `a` まで粗くなってしまう。
これが「統一は片方向の代入も双方向に見なす」の中身だ。

精度の軸は他にもある。**文脈感度**（context-sensitivity）は「同じ関数でも呼び出し文脈ごとに
別々に解析する」か（例：`id(x)` を、呼ぶ場所ごとに別々に解析して戻り値を追う）、
**フロー感度**（flow-sensitivity）は「文の順序を考慮する」か（例：`x = 1; x = 2` を順に見て、最後は `x` が 2 だと知る）。
感度を上げるほど精密になるが、コストは急騰する。文脈感度なら呼び出し文脈の組み合わせの数だけ、
フロー感度ならプログラム点ごとに、別々の解析結果を持たねばならないからだ。実用の解析は、対象規模に応じてこの感度を選ぶ。
JIT のように時間が限られる場面では Steensgaard 流の粗い解析、静的な重武装の解析では Andersen 流以上、
というのが典型だ。

> [!NOTE]
> 言語が参照の書き換え可能性をどれだけ許すかで、別名解析の必要度が変わる。
> C のように任意のポインタ演算を許す言語は別名が悪夢的に難しく、Java/Ruby のように
> 参照が型付きで限定的な言語はまだ扱いやすい。Rust の所有権と借用は、言語設計の側から
> 「別名をコンパイラに保証させる」アプローチだと見ることもできる。
> 別名解析の難しさを、型システムで肩代わりしているわけだ。

## エスケープ解析

三つ目の神器が**[エスケープ解析](#index:エスケープ解析)**（escape analysis）だ。
ある関数の中で作ったオブジェクトが、その関数の外へ**漏れ出す**（escape する）か、
すなわち、戻り値になる、引数経由で渡される、グローバルやフィールドに格納される、
別スレッドから見えるか、を判定する[](#cite:choi1999)。

漏れないと分かれば、二つの強力な最適化ができる。

- **[スカラ置換](#index:スカラ置換)（scalar replacement）**: オブジェクトを作らず、
  そのフィールドを個別のローカル変数（スカラ）として扱う。ヒープ割り当てそのものが消える。
- **スタック割り当て**: ヒープでなくスタックに置く。関数を抜ければ自動で消え、GC の負担が減る。
- **同期の除去**: そのオブジェクトが単一スレッドからしか見えないなら、ロックを外せる。

```ruby
def distance(x1, y1, x2, y2)
  dx = Point.new(x1 - x2, y1 - y2)   # この Point は外に漏れない（ローカルだけ）
  Math.sqrt(dx.x ** 2 + dx.y ** 2)   # dx.x, dx.y を使うだけ
end

# エスケープしないと判明 → スカラ置換でオブジェクトを作らない（概念的に）
def distance(x1, y1, x2, y2)
  dx_x = x1 - x2; dx_y = y1 - y2     # フィールドをローカル変数に
  Math.sqrt(dx_x ** 2 + dx_y ** 2)   # ヒープ割り当てゼロ
end
```

エスケープ解析は[インライン展開](inlining.md)と相乗する。インラインで呼び出し先の本体が
呼び出し元に取り込まれると、それまで「引数として外へ渡る＝エスケープ」と見えていたオブジェクトが、
実は「インラインされた本体の中で使い切られる＝エスケープしない」と分かることが多い。
だから現代の JIT は「インライン → エスケープ解析 → スカラ置換」をセットで回す。

Java HotSpot や GraalVM では、より強力な**部分エスケープ解析**（partial escape analysis）が使われる
[](#cite:stadler2014)。これは「ある経路ではエスケープするが、別の経路ではしない」場合に、
エスケープする経路でだけオブジェクトを実体化（materialize）し、しない経路では割り当てを省く。
制御フローに沿ってエスケープ判定を細かく行うことで、丸ごと判定する古典版より多くの割り当てを消せる。

> [!IMPORTANT]
> エスケープ解析は、本シリーズの GC の本が扱う「ヒープ割り当てそのものを減らす」最適化だ。
> GC の仕組みには立ち入らないが、「**割り当てを最適化で消せれば、GC の仕事自体が減る**」
> という関係は重要だ。最速のメモリ管理は「割り当てないこと」であり、それを可能にするのが
> エスケープ解析である。

## 手続き間定数伝播と要約

別名やエスケープのような「メモリの解析」だけでなく、値の解析も関数をまたいで行える。
**手続き間定数伝播**（interprocedural constant propagation; IPCP）は、
「ある関数がつねに定数引数で呼ばれる」なら、その引数を定数として関数内に伝播する。
`f` が常に `f(0)` で呼ばれるなら、`f` の本体を `0` で特殊化できる。

手続き間解析を毎回プログラム全体で回すのは高くつくので、各関数の効果を**要約（summary）**に
まとめて使い回すのが定石だ。「この関数は引数 0 を受け取ると定数 1 を返し、副作用はなく、
第 2 引数をエスケープさせる」といった要約を関数ごとに一度計算し、呼び出し点ではその要約を
当てはめるだけにする。これにより、関数の中身を毎回展開せずに手続き間の事実を伝えられる。
要約ベースの解析は、[全プログラム最適化](whole-program-optimization.md)や LTO の基盤技術だ。

## まとめ

- コールグラフは「誰が誰を呼ぶか」の地図。精度は間接呼び出し（動的ディスパッチ・関数ポインタ）の解決精度で決まる。
- 別名解析は「同じ場所を指すか」を解く。Andersen 流（精密・重い）と Steensgaard 流[](#cite:steensgaard1996)（粗い・ほぼ線形）が代表。文脈感度とフロー感度で精度とコストを調整する。
- エスケープ解析[](#cite:choi1999)は「外に漏れるか」を解き、スカラ置換やスタック割り当て、同期除去を可能にする。インラインと相乗し、部分エスケープ解析[](#cite:stadler2014)でさらに強くなる。
- 関数の効果を要約にまとめると、手続き間の事実（定数、副作用、エスケープ）を全体に効率よく伝えられる。

これらの解析がそろうと、いよいよ「プログラム全体を見渡せるとき」にしかできない最適化が開ける。
次章は脱仮想化、LTO、部分評価という、視野の広さが質に変わる全プログラム最適化だ。
