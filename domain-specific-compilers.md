# ドメイン特化コンパイラ — 最適化が「どこでも」起きる時代

本書はここまで、C や Ruby のような汎用言語を機械語へ落とす過程の最適化を見てきた。
だが 2010 年代以降、最適化のもっとも熱い主戦場は別のところへ移った——**特定領域に特化したコンパイラ**だ。
深層学習のテンソル演算、データベースのクエリ、FPGA のハードウェア記述。
これらは「汎用言語の関数」ではないが、いずれも**高レベルの意図を、効率的な低レベル実装へ翻訳する**という、
まぎれもないコンパイルの問題だ。そして本書で見てきた道具——依存解析、ループ変換、融合、
[equality saturation](equality-saturation.md)、[検証](superoptimization.md)、[学習](ml-and-future.md)——が、
ここで一斉に再利用されている。

この章は、その「コンパイラがどこにでもある時代」を概観する。共通する設計原理が一つある——
**高レベルの意味を、早く捨てない**。一つの低レベル IR（LLVM IR など）に急いで落とすと、
テンソル融合やループタイリングといった領域特化の最適化機会が失われてしまう。
だから現代の基盤は、高レベルの抽象から低レベルへ**段階的に下げる**。[前章で触れた](ml-and-future.md)
MLIR[](#cite:lattner2021)の「複数の方言を共存させ、最適化しながら降りる」設計は、まさにこの原理の結晶だ。

## アルゴリズムとスケジュールを分ける

この分野を方向づけた一つのアイデアが、画像処理 DSL **Halide**[](#cite:ragankelley2013halide)の
**アルゴリズムとスケジュールの分離**だ。従来のプログラムは「何を計算するか」と
「どの順で計算し、中間結果をどこに置くか」を一つのコードに混ぜていた。Halide はこれを切り離す。

- **アルゴリズム**: 各出力が入力からどう決まるか。順序も記憶域も指定しない。
- **スケジュール**: どの順で回すか、何を並列化・タイリング・融合するか、中間結果を保持するか再計算するか。

```
# 概念: 同じアルゴリズムに、別々のスケジュールを与える
algorithm:  blur(x,y) = (in(x-1,y) + in(x,y) + in(x+1,y)) / 3   # 何を計算するか
schedule A: 素直に全部計算してから次段へ      # 局所性が悪い
schedule B: タイルごとに縦横を融合し並列化    # 同じ計算、桁違いに速い
```

素朴な実装と、並列性・局所性を尽くした実装の差は、しばしば**一桁**にもなる[](#cite:ragankelley2018halidecacm)。
アルゴリズムを書き換えずにこの差を引き出せる——これが分離の威力だ。
深層学習コンパイラ **TVM**[](#cite:chen2018tvm)も、この発想をテンソル計算へ受け継いでいる
（同時期の Glow[](#cite:rotem2018glow)は、高レベル IR で領域特化最適化を、低レベル IR でメモリ最適化を行う
二段構えを取った——どの抽象で何を最適化するかという、この章を貫く問いの初期の答えだ）。

## スケジュールを自動で探す

スケジュールを分離できたなら、次の問いは自然だ——**それを人手で書く必要があるのか**。
スケジュール空間は巨大で、最良の一点を人間が見つけるのは難しい。そこで**自動スケジューラ**が登場する。

Adams らの Halide 自動スケジューラ[](#cite:adams2019autoscheduler)は、ビーム探索で膨大なスケジュール候補を
探り、**何十万ものランダムプログラムで訓練した[コストモデル](ml-and-future.md)**で各候補を評価する。
これは「平均して人間の専門家を上回った最初の自動スケジューラ」だった。
[前章で触れた](ml-and-future.md) **Ansor**[](#cite:zheng2020ansor)も、テンプレートに縛られない探索と
学習コストモデルでテンソルプログラムを自動チューニングする。GPU 向けの自動化[](#cite:anderson2021gpuauto)や、
異種ハードウェアへ広げた FlexTensor[](#cite:zheng2020flextensor)もこの系譜だ。

逆に、自動化を信用せず**人間が書くスケジュールの正しさを保証する**方向もある。
**Exo**[](#cite:ikarashi2022exo)は、ハードウェア固有の最適化をユーザ側のライブラリに外部化し、
スケジュール変換が元のアルゴリズムと等価であることを効果解析で保証する——
「人がスケジュールを書く」と「壊さない」を両立させる試みだ。

## テンソルグラフを書き換える

スケジュール（一つの演算子をどう実装するか）の上には、**グラフ**（演算子どうしをどうつなぐか）の層がある。
ここでの最適化は、本書の[ピープホール](peephole.md)や[値番号付け](local-optimization.md)の
テンソル版だ——等価な部分グラフへの書き換えと、演算子の融合。

- **TASO**[](#cite:jia2019taso)は、演算子の仕様から書き換え規則を**自動生成**し、[定理証明器（Z3）で正しさを検証](superoptimization.md)してから、コストにもとづく探索で適用する。人手の規則集を、合成と検証で置き換えた。
- **PET**[](#cite:wang2021pet)は、**部分的にしか等価でない**変換まで許し、ずれを後から補正する。厳密な等価に縛られない大胆な書き換えで、さらなる高速化を得る。
- **Tensat**[](#cite:yang2021tensat)は、この書き換えを [equality saturation](equality-saturation.md) で行う——逐次探索の TASO に対し、すべての書き換えを同時に保持して最良を選ぶ（[前章](equality-saturation.md)で見た発想のテンソル版だ）。

そして演算子融合の主目的は、[ループ最適化の章で触れた](loop-optimization.md)とおり**メモリトラフィックの削減**にある。
中間テンソルを主記憶へ書き戻さず、オンチップに載せたまま次の演算へ渡す。
Welder[](#cite:shi2023welder)はこの「メモリアクセスをタイルグラフとして最適化する」視点を前面に出し、
Roller[](#cite:zhu2022roller)は探索でなく構築でカーネルを数秒で生成し、
Hidet[](#cite:ding2023hidet)はスケジュールをプログラムに埋め込む粒度の細かい方式を取る。

## GPU カーネルと多面体の復権

より低い層、GPU カーネルそのものを書くための DSL も育った。**Triton**[](#cite:tillet2019triton)は、
スレッドでなく**タイル**を単位にしたプログラミングモデルで、ブロック内の並列化やメモリ合体を
コンパイラに任せる。いまや深層学習カーネルの主要な記述手段の一つだ。

[ループ最適化で触れた多面体モデル](loop-optimization.md)も、この文脈で復権した。
**Tiramisu**[](#cite:baghdadi2019tiramisu)は多面体スケジューリングを明示的な言語で扱い、
**Polygeist**[](#cite:moses2021polygeist)は C を MLIR の多面体表現へ「持ち上げ」て、
既存コードに多面体最適化を適用する。関数型の書き換え戦略でポータブルな性能を狙う
Lift/RISE[](#cite:steuwer2017lift)[](#cite:hagedorn2020rise)や、入れ子並列を備えた純粋関数型 GPU 言語
Futhark[](#cite:henriksen2017futhark)、既存コードに多面体並列化を透過的に施す Polly-ACC[](#cite:grosser2016pollyacc)もある。

## コンパイルの新しい客

テンソルだけではない。「高レベルの記述を効率的な低レベルへ落とす」という構図は、思わぬ領域で再発見された。

**データベースのクエリ**もコンパイルされる。古典的なクエリ実行は、演算子をまたぐたびに仮想呼び出しが走る
インタプリタ（Volcano モデル）だった。Neumann は、これを**データ中心の push 型**に組み替え、
クエリプランを LLVM IR へ**コンパイル**することで、手書き C に迫る速度を得た[](#cite:neumann2011query)。
さらに生成的な手法（軽量モジュラーステージング[](#cite:rompf2010lms)を使う LegoBase[](#cite:klonatos2014legobase)）は、
「クエリエンジンとは、クエリを引数に[部分評価](whole-program-optimization.md)されたインタプリタである」という
[フタムラ射影](whole-program-optimization.md)の視点を明確にした[](#cite:tahboub2018revisited)。
**Weld**[](#cite:palkar2017weld)は、pandas や NumPy など**ライブラリの境界を越えて**共通 IR 上でループ融合し、
データ解析アプリ全体を最適化する[](#cite:palkar2018weld)。Apache Spark の whole-stage code generation[](#cite:armbrust2015sparksql)は、
これを本番分散処理に持ち込んだ例だ（コンパイル型とベクトル化型の優劣は今も議論が続く[](#cite:kersten2018everything)）。

**ハードウェアそのもの**もコンパイル対象だ。高位合成（HLS）は C やDSL を回路（FPGA・ASIC）へ落とす。
Calyx[](#cite:nigam2021calyx)は、ソフトウェア風の制御フローとハードウェア風の構造を一つの中間言語に
併せ持ち、両方の情報を要する新しい最適化を可能にする。Dahlia[](#cite:nigam2020dahlia)は、
ハードウェア資源の同時使用を**時間感応アフィン型**で禁じ、HLS の「パラメータをいじったら逆に遅くなる」
予測不能性を型で抑える。Spatial[](#cite:koeplinger2018spatial)はアクセラレータ記述の言語であり、
[CompCert](superoptimization.md)の上に**形式検証された HLS**(Vericert[](#cite:herklotz2021vericert))まで現れた。
ハードウェアコンパイラの基盤を MLIR 流の方言群で築く CIRCT[](#cite:circt)もあり、
[ソフトウェアの MLIR](ml-and-future.md)と同じ「段階的に下げる」方法論が回路設計にも持ち込まれている。

極めつけは**自動微分**だ。深層学習に不可欠な勾配計算を、ライブラリでなく**コンパイラ変換**として実装する。
**Enzyme**[](#cite:moses2020enzyme)は、最適化済みの LLVM IR を微分する——
**最適化してから微分する**ことで、微分してから最適化するより約 4.2 倍速い勾配を生む。
GPU カーネルの微分[](#cite:moses2021enzymegpu)にも届いている。源流をたどれば、Fortran/C を
ソース変換する Tapenade[](#cite:hascoet2013tapenade)や、Julia の [SSA](ssa-optimization.md) を直接微分する
Zygote[](#cite:innes2018zygote)があり、その理論的な核は驚くほど単純な圏論的構造に整理できる[](#cite:elliott2018ad)。
「微分」さえコンパイラの最適化問題になった、というわけだ。

> [!NOTE]
> これらに共通するのは、本章冒頭の原理——**高レベルの意味を早く捨てない**——だ。
> テンソルの形、クエリの関係代数、回路の構造、微分すべき計算。これらは低レベル IR に落ちた瞬間に
> 見えなくなる。だから現代の基盤は、領域固有の高レベル表現で最適化を済ませてから、段階的に下げる。
> MLIR の方言、Halide のスケジュール、Calyx の制御＋構造——形は違えど、狙いは同じだ。

## まとめ

- 最適化の主戦場は、汎用言語の機械語生成から**領域特化コンパイラ**へ広がった。共通原理は「高レベルの意味を早く捨てない／段階的に下げる」。
- Halide[](#cite:ragankelley2013halide)の**アルゴリズムとスケジュールの分離**が分野を方向づけた。スケジュールは学習[](#cite:adams2019autoscheduler)で自動探索され、Exo[](#cite:ikarashi2022exo)のように正しさを保証する方向もある。
- テンソルグラフは、規則合成＋検証（TASO[](#cite:jia2019taso)）・部分等価変換（PET[](#cite:wang2021pet)）・equality saturation（Tensat[](#cite:yang2021tensat)）で書き換えられ、融合でメモリトラフィックを削る[](#cite:shi2023welder)。
- クエリ[](#cite:neumann2011query)・データ解析[](#cite:palkar2017weld)・高位合成[](#cite:nigam2021calyx)・自動微分[](#cite:moses2020enzyme)——「コンパイル」はあらゆる領域へ広がり、本書の道具がそこで再利用されている。

次章（最終章）では、こうした多彩な最前線を貫く一本の糸——人手のヒューリスティクスの限界を、
探索・学習・検証で越えるという動機——を確かめ、本書を締めくくる。
