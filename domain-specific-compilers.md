# ドメイン特化コンパイラ — 最適化が「どこでも」起きる時代

本書はここまで、C や Ruby のような汎用言語を機械語へ落とす過程の最適化を見てきた。
だが 2010 年代以降、最適化のもっとも熱い主戦場は別のところへ移った——**特定領域に特化したコンパイラ**だ。
深層学習のテンソル演算、データベースのクエリ、FPGA のハードウェア記述。
これらは「汎用言語の関数」ではないが、いずれも**高レベルの意図を、効率的な低レベル実装へ翻訳する**という、
まぎれもないコンパイルの問題だ。そして本書で見てきた道具——依存解析、ループ変換、融合、
[equality saturation](equality-saturation.md)、[検証](superoptimization.md)、[学習](ml-and-future.md)——が、
ここで一斉に再利用されている。

この章は、その「コンパイラがどこにでもある時代」を概観する。共通する設計原理が一つある——
**高レベルの意味を、早く捨てない**。一つの低レベル IR（LLVM IR など）に急いで落とすと、
テンソル融合やループタイリングといった領域特化の最適化機会が失われてしまう。
だから現代の基盤は、高レベルの抽象から低レベルへ**段階的に下げる**。[次章で扱う](ml-and-future.md)
MLIR[](#cite:lattner2021)の「複数の方言を共存させ、最適化しながら降りる」設計は、まさにこの原理の結晶だ。

この章は固有名が多く出るが、**一つひとつを覚える必要はない**。「どんな発想が、どの領域で再利用されているか」
という流れだけ掴めば十分だ。各システムの名前は、後で原典を当たるときの索引だと思ってほしい。

## アルゴリズムとスケジュールを分ける

この分野を方向づけた一つのアイデアが、画像処理 DSL **Halide**[](#cite:ragankelley2013halide)の
**アルゴリズムとスケジュールの分離**だ。従来のプログラムは「何を計算するか」と
「どの順で計算し、中間結果をどこに置くか」を一つのコードに混ぜていた。Halide はこれを切り離す。

- **アルゴリズム**: 各出力が入力からどう決まるか。順序も記憶域も指定しない。
- **スケジュール**: どの順で回すか、何を並列化・タイリング・融合するか、中間結果を保持するか再計算するか。

```
# 概念: 同じアルゴリズムに、別々のスケジュールを与える
algorithm:  blur(x,y) = (in(x-1,y) + in(x,y) + in(x+1,y)) / 3   # 何を計算するか
schedule A: 素直に全部計算してから次段へ      # 局所性が悪い
schedule B: タイルごとに縦横を融合し並列化    # 同じ計算、桁違いに速い
```

素朴な実装と、並列性・局所性を尽くした実装の差は、しばしば**一桁**にもなる[](#cite:ragankelley2018halidecacm)。
アルゴリズムを書き換えずにこの差を引き出せる——これが分離の威力だ。
深層学習コンパイラ **TVM**[](#cite:chen2018tvm)も、この発想をテンソル計算へ受け継ぎ、現代の AI コンパイラの礎になった。

## スケジュールを自動で探す

スケジュールを分離できたなら、次の問いは自然だ——**それを人手で書く必要があるのか**。
スケジュール空間は巨大で、最良の一点を人間が見つけるのは難しい。そこで**自動スケジューラ**が登場する。

Adams らの Halide 自動スケジューラ[](#cite:adams2019autoscheduler)は、候補を探索木で広く探り、
**何十万ものランダムプログラムで訓練した[コストモデル](ml-and-future.md)**で各候補の速さを見積もる。
これは「平均して人間の専門家を上回った最初の自動スケジューラ」だった。
同じ「学習したコストモデルで探索する」発想は、テンソル計算の自動チューニング（Ansor[](#cite:zheng2020ansor) など）にも広がっている。

逆に、自動化を信用せず**人間が書くスケジュールの正しさを保証する**方向もある。
**Exo**[](#cite:ikarashi2022exo)は、スケジュール変換が元のアルゴリズムと等価であることを解析で保証する——
「人がスケジュールを書く」と「壊さない」を両立させる試みだ。

## テンソルグラフを書き換える

スケジュール（一つの演算子をどう実装するか）の上には、**グラフ**（演算子どうしをどうつなぐか）の層がある。
ここでの最適化は、本書の[ピープホール](peephole.md)や[値番号付け](local-optimization.md)の
テンソル版だ——等価な部分グラフへの書き換えと、演算子の融合。

- **TASO**[](#cite:jia2019taso)は、演算子の仕様から書き換え規則を**自動生成**し、
  [定理証明器（Z3）で正しさを検証](superoptimization.md)してから、コストにもとづく探索で適用する。
  人手の規則集を、合成と検証で置き換えた（厳密には等価でない変換まで許して後で補正する、より大胆な手もある[](#cite:wang2021pet)）。
- **Tensat**[](#cite:yang2021tensat)は、この書き換えを [equality saturation](equality-saturation.md) で行う——
  逐次探索の TASO に対し、すべての書き換えを同時に保持して最良を選ぶ（[equality saturation の章](equality-saturation.md)で見た発想のテンソル版だ）。

そして演算子融合の主目的は、[ループ最適化の章で触れた](loop-optimization.md)とおり**メモリトラフィックの削減**にある。
中間テンソルを主記憶へ書き戻さず、オンチップに載せたまま次の演算へ渡す——
この「メモリアクセスを最適化の主役に据える」視点を前面に出したのが Welder[](#cite:shi2023welder)だ。

## GPU カーネルと多面体の復権

より低い層、GPU カーネルそのものを書くための DSL も育った。**Triton**[](#cite:tillet2019triton)は、
スレッドでなく**タイル**（小さなデータの塊）を単位にしたプログラミングモデルで、
ブロック内の並列化やメモリアクセスの最適化をコンパイラに任せる。いまや深層学習カーネルの主要な記述手段の一つだ。

[多面体モデル](loop-optimization.md)——ループの反復を格子点とみなし、依存を壊さない範囲で順序を変える理論——も、
この文脈で復権した。たとえば **Polygeist**[](#cite:moses2021polygeist)は、ふつうの C を MLIR の多面体表現へ
「持ち上げて」、既存コードにそのまま多面体最適化を適用する。

> [!NOTE]
> **発展**：テンソル・GPU コンパイラの広がり。
> 自動スケジューリングは GPU 向け[](#cite:anderson2021gpuauto)や異種ハードウェア（FlexTensor[](#cite:zheng2020flextensor)）へ広がった。
> 融合・カーネル生成では、探索でなく構築で数秒生成する Roller[](#cite:zhu2022roller)、
> スケジュールをプログラムに埋め込む Hidet[](#cite:ding2023hidet)、二段 IR の Glow[](#cite:rotem2018glow)がある。
> 多面体系には Tiramisu[](#cite:baghdadi2019tiramisu)、関数型の書き換え戦略で性能を出す Lift/RISE[](#cite:steuwer2017lift)[](#cite:hagedorn2020rise)、
> 純粋関数型 GPU 言語 Futhark[](#cite:henriksen2017futhark)、C を透過的に並列化する Polly-ACC[](#cite:grosser2016pollyacc)など。

## コンパイルの新しい客

テンソルだけではない。「高レベルの記述を効率的な低レベルへ落とす」という構図は、思わぬ領域で再発見された。
代表例を一つずつ見よう。

**データベースのクエリ**もコンパイルされる。古典的なクエリ実行は、演算子をまたぐたびに関数呼び出しが走る
インタプリタだった。Neumann は、これをやめてクエリを丸ごと機械語へ**コンパイル**し、
手書き C に迫る速度を得た[](#cite:neumann2011query)。
**Weld**[](#cite:palkar2017weld)は、pandas や NumPy など**ライブラリの境界を越えて**共通の中間表現でループ融合し、
データ解析アプリ全体を最適化する[](#cite:palkar2018weld)。

**ハードウェアそのもの**もコンパイル対象だ。高位合成（HLS）は C や DSL を回路（FPGA・ASIC）へ落とす。
**Calyx**[](#cite:nigam2021calyx)は、ソフトウェア風の制御フローとハードウェア風の構造を一つの中間言語に
併せ持ち、両方の情報が要る最適化を可能にする。

極めつけは**自動微分**だ。深層学習に不可欠な勾配計算を、ライブラリでなく**コンパイラ変換**として実装する。
**Enzyme**[](#cite:moses2020enzyme)は、最適化済みの LLVM IR を微分する——
**最適化してから微分する**ことで、微分してから最適化するより約 4.2 倍速い勾配を生む。
「微分」さえコンパイラの最適化問題になった、というわけだ。

> [!NOTE]
> **発展**：各領域のさらなる広がり。
> クエリ側では、「クエリエンジンとは、クエリで[部分評価](whole-program-optimization.md)されたインタプリタだ」という
> [フタムラ射影](whole-program-optimization.md)の視点[](#cite:tahboub2018revisited)（LegoBase[](#cite:klonatos2014legobase)[](#cite:rompf2010lms)）、
> 本番分散処理での実装（Spark[](#cite:armbrust2015sparksql)）、コンパイル型 vs ベクトル化型の論争[](#cite:kersten2018everything)がある。
> HLS では、資源の同時使用を型で禁じて性能を予測可能にする Dahlia[](#cite:nigam2020dahlia)、
> アクセラレータ言語 Spatial[](#cite:koeplinger2018spatial)、[CompCert](ml-and-future.md) 上の検証付き HLS（Vericert[](#cite:herklotz2021vericert)）、
> MLIR 流の回路基盤 CIRCT[](#cite:circt)。自動微分では GPU カーネルの微分[](#cite:moses2021enzymegpu)、
> 源流の Tapenade[](#cite:hascoet2013tapenade)・Zygote[](#cite:innes2018zygote)、その理論的整理[](#cite:elliott2018ad)など。

> [!NOTE]
> これらに共通するのは、本章冒頭の原理——**高レベルの意味を早く捨てない**——だ。
> テンソルの形、クエリの関係代数、回路の構造、微分すべき計算。これらは低レベル IR に落ちた瞬間に
> 見えなくなる。だから現代の基盤は、領域固有の高レベル表現で最適化を済ませてから、段階的に下げる。
> MLIR の方言、Halide のスケジュール、Calyx の制御＋構造——形は違えど、狙いは同じだ。

## まとめ

- 最適化の主戦場は、汎用言語の機械語生成から**領域特化コンパイラ**へ広がった。共通原理は「高レベルの意味を早く捨てない／段階的に下げる」。
- Halide の**アルゴリズムとスケジュールの分離**が分野を方向づけた。スケジュールは学習で自動探索でき、Exo のように正しさを保証する方向もある。
- テンソルグラフは、規則合成＋検証（TASO）・equality saturation（Tensat）で書き換えられ、融合でメモリトラフィックを削る（Welder）。
- クエリ・データ解析・高位合成・自動微分——「コンパイル」はあらゆる領域へ広がり、本書の道具がそこで再利用されている。

次章（最終章）では、こうした多彩な最前線を貫く一本の糸——人手のヒューリスティクスの限界を、
探索・学習・検証で越えるという動機——を確かめ、本書を締めくくる。
