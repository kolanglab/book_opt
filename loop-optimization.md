# ループ最適化

プログラムの実行時間は、ごく一部のコードに集中する。経験則として「90% の時間は 10% のコードで
費やされる」とよく言われ、その 10% はたいていループだ。だからループ最適化は最適化の主戦場であり、
ここで紹介する技法はどれも「ループの中で何度も実行される命令を、いかに減らす、軽くする、並列化するか」
に尽きる[](#cite:aho2006)[](#cite:muchnick1997)。

この章では、ループをまず構造として認識する方法から始め、ループ不変式の移動、帰納変数の強度低減、
アンローリング、ループ融合と分配、そしてベクトル化まで、ループ専用の最適化を一つずつ Ruby で確かめる。

## ループを見つける

最適化の前に、CFG のどこがループかを機械的に特定する必要がある。鍵は**[逆辺](#index:逆辺)**
（back edge）だ。辺 `n → h` のうち、**ヘッダ h が n を支配する**ものを逆辺と呼ぶ
（[支配関係](intermediate-representation.md)を思い出そう）。逆辺は「ループの末尾から先頭へ戻る矢印」にあたる。

逆辺 `n → h` が定まると、その**[自然ループ](#index:自然ループ)**（natural loop）は
「h を通らずに n に到達できるブロック全体＋h」として求まる。h が**ループヘッダ**、
ループに入る唯一の入口だ。ヘッダの直前に、ループに入る経路だけを集めた**プレヘッダ**
（preheader）という空ブロックを挿入しておくと、「ループの外に命令を追い出す」最適化の置き場所になる。

```ruby
# back_edge: [n, h] （h が n を支配）から自然ループの本体集合を求める
def natural_loop(n, h, preds)
  body = Set[h, n]
  stack = [n]
  until stack.empty?
    m = stack.pop
    preds[m].each do |p|
      unless body.include?(p)    # h を通らずに遡れる範囲を集める
        body << p
        stack << p
      end
    end
  end
  body
end
```

ループには入れ子（ループの中のループ）があり、外側ほど実行回数は少ない。
最適化は内側の熱いループを優先する。以降の技法は、すべてこの「自然ループ＋プレヘッダ」の上で動く。

## ループ不変式の移動（LICM）

ループの中で、**毎回同じ値を計算している**式がある。

```ruby
i = 0
while i < n
  x = a * b      # a も b もループ内で変わらない → 毎回同じ。無駄
  arr[i] = x + i
  i += 1
end
```

`a * b` はループ内で `a` も `b` も再代入されないなら、**ループ不変**（loop-invariant）だ。
これをプレヘッダに追い出せば、n 回の乗算が 1 回で済む。これが**[ループ不変式の移動](#index:ループ不変式の移動)**
（loop-invariant code motion; LICM）だ[](#cite:muchnick1997)。LICM と[共通部分式除去](local-optimization.md)を合流まで含めて統一的に扱う一般化が**部分冗長除去**（partial redundancy elimination; PRE）で、その計算を「早すぎず遅すぎない」最適位置に置く古典的手法が lazy code motion[](#cite:knoop1992) である。

```ruby
i = 0
t = a * b        # プレヘッダへ巻き上げ（hoisting）
while i < n
  arr[i] = t + i
  i += 1
end
```

ある式がループ不変だと判定する条件は再帰的だ。「オペランドがすべて、(1) 定数か、
(2) 定義がループ外にあるか、(3) それ自身がループ不変な式の結果」であること。
SSA だとこの判定はとても素直に書ける。各定義を見て、オペランドの定義がループ本体の外にあるかを
辿るだけだ。

```ruby
# loop_body: ループ内ブロックの集合。def_block: 変数 => 定義ブロック。
def loop_invariant?(instr, loop_body, def_block, invariant)
  instr.args.all? do |a|
    constant?(a) ||
      !loop_body.include?(def_block[a]) ||   # 定義がループ外
      invariant.include?(a)                  # 既に不変と判明
  end
end
```

> [!WARNING]
> LICM の正しさには見落としがちな罠がある。第一に、**副作用**のある式（メソッド呼び出しなど）は、
> 不変でも勝手に巻き上げてはいけない（ループが 0 回しか回らないとき、本来実行されないはずの計算が
> 走ってしまう）。第二に、**例外やゼロ除算**を起こしうる式（`a / b`）は、ループが実行されないケースで
> 偽の例外を出しうる。安全には「移動先がループ内の全使用を支配し、かつループが必ず 1 回以上回る
> （または式が安全に投機実行できる）」ことを確かめる。速くするための移動が意味を壊しては本末転倒だ。

## 帰納変数と強度低減

ループの花形最適化が**[帰納変数](#index:帰納変数)**（induction variable）の**強度低減**だ。
ループカウンタ `i` のように、毎回**一定量ずつ増える**変数を帰納変数（基本帰納変数）と呼ぶ。
そして `i` の一次式 `c1 * i + c2` で表せる変数（`j = 4 * i` など）を**派生帰納変数**と呼ぶ。

配列アクセス `arr[i]` は、内部的には `base + i * 4`（要素 4 バイト）というアドレス計算になる。
ループのたびに乗算 `i * 4` が走るのは無駄だ。`i` が毎回 1 増えるなら、`i * 4` は毎回 4 増える。
だから乗算をやめ、**前回の値に 4 を足す加算**に置き換えられる。これが
[強度低減](peephole.md)のループ版で、ループ最適化で最も効果が大きいものの一つだ[](#cite:allen1972)[](#cite:muchnick1997)。

```ruby
# 変換前: 毎回 i*4 を乗算
i = 0
while i < n
  addr = base + i * 4    # 乗算
  store addr, 0
  i += 1
end

# 変換後: 加算だけ（addr 自身が帰納変数になった）
i = 0
addr = base             # 初期値
while i < n
  store addr, 0
  addr += 4             # 強度低減: i*4 の差分を足す
  i += 1
end
```

さらにここまで来ると、`i` 自身が「`i < n` の判定」以外で使われていないことがある。
その場合、ループの終了条件を `addr < base + n*4` に書き換えれば、`i` を丸ごと消せる。
これが**帰納変数の除去**（induction variable elimination）だ。
強度低減で派生変数を作り、不要になった元の帰納変数を消す、という二段構えで配列ループは大きく軽くなる。

派生帰納変数の検出は、SSA 上で「φ で自分自身＋定数になっている変数（基本帰納変数）」を見つけ、
その線形結合を辿る、という形で実装できる。[scalar evolution](#index:scalar evolution)（LLVM の SCEV）は、
こうした帰納変数の閉じた式を体系的に求める近代的な仕組みで、ベクトル化やループ境界の解析にも使われる。

## ループアンローリング

**[ループアンローリング](#index:ループアンローリング)**（loop unrolling; ループ展開）は、
ループ本体を数回ぶん複製して、1 周あたりの反復回数を増やす変換だ。

```ruby
# 4 回展開: ループ制御（i<n, i+=1）の回数が 1/4 に
i = 0
while i + 4 <= n
  body(i); body(i+1); body(i+2); body(i+3)
  i += 4
end
while i < n        # 端数処理（残り 0〜3 回）
  body(i); i += 1
end
```

狙いは三つある。第一に、ループの**制御オーバーヘッド**（カウンタ更新と条件分岐）を 1/4 に減らす。
第二に、展開で並んだ独立な命令が、後述のベクトル化や[命令スケジューリング](instruction-scheduling.md)で
パイプラインを埋めやすくなる。第三に、分岐が減ることで分岐予測ミスが減る。

代償はコードサイズの増加だ。展開しすぎると命令キャッシュを圧迫し、かえって遅くなる。
だから展開回数は本体の大きさと相談して決める（典型的なフェーズ順序とトレードオフの一例だ）。
「端数処理」の後始末（n が 4 の倍数でないとき）も忘れてはいけない。

## ループ融合、分配、交換

ループの**構造そのもの**を組み替える最適化群もある。

- **[ループ融合](#index:ループ融合)（loop fusion）**: 同じ範囲を回る隣り合った 2 ループを 1 つにまとめる。
  配列を 2 回なめる代わりに 1 回で済み、キャッシュ局所性が上がる。
- **ループ分配（loop fission/distribution）**: 逆に 1 ループを 2 つに割る。
  片方だけベクトル化できる、依存を断ち切れる、といった場合に効く。融合の逆向きだ。
- **ループ交換（loop interchange）**: 入れ子ループの内と外を入れ替える。
  多次元配列を「メモリ上で連続な向き」になめるよう直すと、キャッシュミスが激減する。

> [!NOTE]
> 同じ「中間データを作らず一度に流す」発想は、関数型言語では**融合（fusion）／脱森林化（deforestation）**
> と呼ばれ、書き換え規則として実現される。`map f . map g` を `map (f . g)` に畳む短絡融合[](#cite:gill1993deforestation)や、
> リスト処理を中間リストなしのループに変える stream fusion[](#cite:coutts2007streamfusion)が代表で、
> GHC ではこれらを[ピープホール的な書き換え規則](peephole.md)としてユーザが宣言できる[](#cite:peytonjones2001rules)。
> 命令型のループ融合と、関数型の deforestation は、「メモリ往復を消す」同じ目標への別表現だ。

```ruby
# 融合前: 配列を 2 回なめる
n.times { |i| b[i] = a[i] * 2 }
n.times { |i| c[i] = b[i] + 1 }
# 融合後: 1 回でまとめて処理（a[i]→b[i]→c[i] がキャッシュに乗ったまま進む）
n.times { |i| b[i] = a[i] * 2; c[i] = b[i] + 1 }
```

これらの変換は強力だが、正しさには**データ依存**の解析が要る。融合した結果、
本来「先に全部 b を埋めてから c を作る」前提が崩れて答えが変わってはいけない。
2 つの反復が同じ配列要素に読み書きで触れる**ループ運搬依存**（loop-carried dependence）があると、
融合や交換が不正になりうる。ループ間の依存を厳密に扱う理論が**多面体モデル**
（[polyhedral model](#index:多面体モデル)）だ。ループの反復を多次元の格子点とみなし、依存を壊さない範囲で
実行順序を変換する。Pluto[](#cite:bondhugula2008pluto)は、この変換を並列性と局所性が同時に最大になるよう
自動で選び、タイリングまで自動化した代表例だ。理論は重いが、自動並列化と自動ベクトル化の基盤として、
GCC、LLVM、MLIR など主要なコンパイラ基盤に採り入れられている[](#cite:verdoolaege2010isl)[](#cite:verdoolaege2013ppcg)[](#cite:lattner2021)。

## ベクトル化

最後に**[自動ベクトル化](#index:自動ベクトル化)**（automatic vectorization）。
現代の CPU は SIMD（single instruction, multiple data; 一命令で複数データ）命令を持ち、
たとえば 4 つの `float` の加算を 1 命令でこなす。スカラのループを SIMD 命令に変換するのがベクトル化だ。

```ruby
# スカラ: 1 要素ずつ
n.times { |i| c[i] = a[i] + b[i] }
# ベクトル（概念図）: 4 要素まとめて。SIMD 加算 1 回が 4 回ぶんの仕事をする
i = 0
while i + 4 <= n
  c[i,4] = simd_add(a[i,4], b[i,4])   # 1 命令で 4 要素加算
  i += 4
end
# 端数はスカラで処理
```

ベクトル化が成立する条件は、**反復間に依存がない**ことだ。`c[i]` の計算が `c[i-1]` の結果を
必要としない、などである。これも多面体モデルやループ運搬依存の解析が支える。
依存があってもベクトル化できる特殊形（リダクション、すなわち総和や最大値）には専用の扱いがある。

実務では、自動ベクトル化が効かないとき、プログラマが SIMD 組み込み関数を直接書いたり、
`#pragma omp simd` のようなヒントを与えたりする。「コンパイラはどこまで自動でやれるか、
どこからは人間のヒントが要るか」の境界が、ベクトル化研究のフロンティアだ。

> [!NOTE]
> **発展**：ベクトル化研究の最前線。
> ループの外で隣り合う独立な文を束ねる **SLP**（superword-level parallelism）も進化した。
> 束ね方を関数全体の整数計画として大域的に解く goSLP[](#cite:mendis2018goslp)、非対称な SIMD 命令まで使えるよう
> SLP を一般化した VeGen[](#cite:chen2021vegen)、ベクトル化を [equality saturation](equality-saturation.md) の探索として解く
> Diospyros[](#cite:vanhattum2021diospyros)と、その規則を ISA 仕様から自動合成する後継 Isaria[](#cite:thomas2024isaria)など。
> ベクトル化もまた、人手のヒューリスティクスから探索、合成、大域最適化へと移りつつある。

> [!NOTE]
> ループアンローリングとベクトル化は相性がよい。展開で並んだ独立命令の塊が、
> そのまま SIMD のレーンに割り振れるからだ。LICM、強度低減、アンローリング、ベクトル化は
> 単独でも効くが、組み合わせると相乗する。フェーズ順序問題の難しさと面白さがここにある。

## メモリ階層を意識する

現代のボトルネックは多くの場合、演算ではなく**メモリ**だ。CPU は演算より桁違いに遅い主記憶を
キャッシュ階層で隠すが、アクセスパターンが悪ければキャッシュは役に立たない。
そこでループとデータの**両方**を、メモリ階層に合わせて作り替える最適化が要る。

- **タイリング（ブロッキング）**: 大きな配列を一度になめると、再利用したいデータがキャッシュから
  追い出される。反復空間を小さなタイルに区切り、タイル内で使い切ってから次へ進むことで、
  キャッシュに載ったまま再利用する。多面体モデル（Pluto[](#cite:bondhugula2008pluto)）が自動で行う中心がこれだ。
- **データ配置変換**: コードでなく**データの並べ方**を変える。構造体の配列（AoS）を配列の構造体（SoA）へ
  組み替えると、同じフィールドだけを連続でなめる処理でキャッシュ効率と SIMD 化が上がる。
  ホットなフィールドとコールドなフィールドを分ける構造体分割も同系統だ[](#cite:chilimbi1999definition)[](#cite:chilimbi1999layout)。
- **ソフトウェアプリフェッチ**: 数反復先で触る番地を、コンパイラがあらかじめ `prefetch` 命令で
  キャッシュへ引き込んでおく。Mowry らが局所性解析にもとづく選択的プリフェッチを確立した[](#cite:mowry1992prefetch)。
- **疎で不規則なアクセス**: 疎行列のように添字が間接参照（`a[idx[i]]`）になると、古典的なループ解析は無力だ。
  疎テンソル代数コンパイラ **TACO**[](#cite:kjolstad2017taco)は、格納形式を抽象化して効率的なループを生成する。

> [!NOTE]
> **発展**：パラメータを機械任せにせず、どのキャッシュサイズでも最適に近づく
> **キャッシュ忘却**（cache-oblivious）アルゴリズム[](#cite:frigo1999cacheoblivious)や、
> SIMD/GPU のメモリ合体を狙ったデータ配置変換[](#cite:henretty2011layout)[](#cite:sung2012layout)もある。

これらは「演算を減らす」最適化ではなく「**メモリトラフィックを減らす**」最適化だ。
のちの[ドメイン特化コンパイラの章](domain-specific-compilers.md)で見るように、演算子融合の主目的も
中間結果を主記憶に書き戻さずに済ませること、つまりメモリ往復を消すことにある。

## まとめ

- ループは逆辺（ヘッダが支配する辺）から自然ループとして特定し、プレヘッダを最適化の置き場にする。
- LICM はループ不変式をプレヘッダに巻き上げる。副作用と例外の安全性に注意。
- 帰納変数の強度低減は乗算を加算に置き換え、配列ループを大きく軽くする。不要になった帰納変数は除去できる。
- アンローリングは制御オーバーヘッドを減らすがコードサイズと引き換え。融合、分配、交換、ベクトル化は依存解析（多面体モデル）が支える。

ここまでで関数内の最適化は出そろった。次の第 III 部では関数の境界を越え、
すべての最適化を呼び込む「[インライン展開](inlining.md)」から、手続き間の世界へ踏み出す。
