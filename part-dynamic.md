# 動的言語の最適化

ここまでの最適化の多くは、変数の型やメソッドの呼び先がコンパイル時に分かることを
暗黙に当てにしていた。だが Ruby・Python・JavaScript・Smalltalk・Self といった
**動的言語**では、`a + b` の `+` が整数加算なのか文字列連結なのか配列結合なのかは、
実行してみるまで分からない。型は変数ではなく値に付き、メソッドは実行時に解決され、
クラスすら実行中に書き換わりうる。静的解析だけでは、ほとんど何も仮定できない。

それでも今日の動的言語が「そこそこ速い」のは、発想を逆転させたからだ。
**コンパイル時に決められないなら、実行時に観測して決めればよい。**
この第 IV 部は、その実行時情報を武器に変える二つの大きな技術系統を扱う。

[動的ディスパッチの章](dynamic-dispatch.md)では、メソッド探索という動的言語最大のコストを、
インラインキャッシュ[](#cite:deutsch1984)・多態インラインキャッシュ[](#cite:holzle1991)・
隠れクラス（shape）で削り取る方法を見る。「前回ここを通った値の形」を覚えておくだけで、
毎回の辞書引きが一回のポインタ比較に化ける。

[投機的最適化の章](speculative-optimization.md)では、観測した型を**仮定**してコードを特殊化し、
仮定が外れたら安全に逃げる（脱最適化[](#cite:holzle1992)）という、現代 JIT の心臓部を扱う。
型フィードバック[](#cite:holzle1994)、トレーシング[](#cite:gal2009)、
基本ブロックバージョニング[](#cite:chevalier2015)——「賭けて、外れたら戻す」という共通の構図を読み取っていく。
