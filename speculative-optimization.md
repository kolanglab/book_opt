# 投機的最適化 — 賭けて、外れたら戻す

[前章](dynamic-dispatch.md)のインラインキャッシュは「観測した型を**覚えて**速くする」ものだった。
この章はもう一歩踏み込む。観測した型を**仮定（speculate）してコードを特殊化し、
仮定が外れたら安全に逃げる**——これが現代の動的言語 JIT の心臓部、
[投機的最適化](#index:投機的最適化)（speculative optimization）だ。

静的最適化は「全実行で正しい」変換しかできなかった。だが投機的最適化は、
「**今回の実行ではこうだった、たぶん次もそうだろう**」という弱い情報を、**取り消せる賭け**として使う。
賭けに勝てば静的言語並みの速さ、負けても正しさは保たれる。この「賭けて、外れたら戻す」という一つの構図が、
型特殊化・脱最適化・トレーシング・基本ブロックバージョニングのすべてを貫いている。

## 型特殊化 — 動的な演算を静的な演算に

Ruby の `a + b` を考える。`+` は `Integer#+` かもしれないし `String#+` かもしれず、
ユーザ定義の `+` かもしれない。だからインタプリタは毎回ディスパッチする。だが
[型フィードバック](dynamic-dispatch.md)[](#cite:holzle1994)——メソッドを受信者の型ごとに特殊化してコンパイルする Self の customization[](#cite:chambers1989) と同じ「型で専用化する」発想だ——が「この `+` は過去 1 万回すべて
`Integer` 同士だった」と告げるなら——**整数加算だと仮定した専用コード**を生成できる。

```ruby
# 一般のディスパッチ版
def add(a, b) = a.send(:+, b)        # 毎回メソッド探索＋ディスパッチ

# Integer を仮定した特殊化版（概念的な疑似コード）
def add_speculated(a, b)
  guard(a.is_a?(Integer) && b.is_a?(Integer))   # ← ガード: 仮定の確認
  machine_int_add(a, b)              # 仮定が正しければ、ただの機械語加算 1 命令
end
```

肝は**[ガード](#index:ガード)**（guard）だ。特殊化コードの入口で「仮定が本当に成り立つか」を
安く確認する。ガードを通れば、あとは型チェックも探索もない、静的言語そのものの機械語が走る。
ガードに失敗したら——特殊化コードは `b` が文字列のケースを**持っていない**。どうするか。
ここで脱最適化が登場する。

## 脱最適化 — 最適化コードから安全に逃げる

**[脱最適化](#index:脱最適化)**（deoptimization）は、最適化コードの実行を途中で打ち切り、
**同じ地点のインタプリタ（または最適化していないコード）に実行を移す**仕組みだ。
Hölzle、Chambers、Ungar が Self で確立した[](#cite:holzle1992)。これがあるからこそ、
JIT は「めったに起きないケースを切り捨てた」大胆な特殊化を安全に行える。

難しいのは**状態の移し替え**だ。最適化コードは、レジスタに値を散らし、
[エスケープしないオブジェクトをスカラ置換](interprocedural-analysis.md)で消し、
[インライン](inlining.md)で複数のフレームを一つに畳んでいるかもしれない。
脱最適化はこの最適化された状態から、インタプリタが期待する素直な状態——
正しいローカル変数の値、正しいスタックフレームの積み重なり——を**再構築**しなければならない。

```
最適化コード（特殊化済み）            インタプリタ
レジスタに散った値          脱最適化     ローカル変数の配列
スカラ置換で消えたオブジェクト  ────▶   オブジェクトを実体化して復元
インラインで畳んだ 3 フレーム            3 つのフレームに展開し直す
```

これを可能にするのが**脱最適化メタデータ**だ。JIT はコードの各「安全な地点」について、
「最適化状態 → インタプリタ状態」の復元マップを記録しておく。脱最適化が起きると、
このマップに従って状態を組み直し、インタプリタへ制御を渡す。
sea of nodes 系の Graal IR は、この投機と脱最適化を IR に第一級で組み込んでいる。

> [!NOTE]
> 「投機して外れたら戻す」が**本当に意味を壊さないか**は、自明ではない。Sourir[](#cite:fluckiger2018sourir)は
> 投機の仮定を IR に明示的に持たせ、定数畳み込み・不要コード除去・インラインが脱最適化の存在下でも
> 正しいことを証明した。後継の CoreJIT[](#cite:barriere2021corejit)は、この投機と脱最適化を備えた
> モデル JIT 全体を Coq で機械検証している。[前章の検証](superoptimization.md)の波は、JIT の心臓部にも届いている。

> [!IMPORTANT]
> 脱最適化は本書の[導入](introduction.md)で立てた正当性の問いの、最も鮮やかな答えだ。
> 投機的最適化は「観測可能な振る舞いを変えない」を、**「外れたら元の正しい実行に戻る」**ことで保証する。
> 速いコードは「正しいかもしれない近道」にすぎず、いつでも「確実に正しい本道」（インタプリタ）へ
> 戻れるようにしてある。賭けと安全網がワンセット——これが動的言語を速く、かつ正しく走らせる秘密だ。

## オンスタック置換（OSR）

脱最適化と対になるのが**[オンスタック置換](#index:オンスタック置換)**（on-stack replacement; OSR）だ。
脱最適化が「最適化コード → インタプリタ」へ降りるのに対し、OSR は逆——
**実行中の関数を、途中から最適化コードに昇格させる**。

なぜ要るか。長いループを 1 回だけ回る関数を考える。ループに入った後で「これはホットだ」と
気づいても、関数の入口はもう通り過ぎている。次に呼ばれるのを待っていては、この実行は永遠に遅いままだ。
OSR は、**ループの途中で**インタプリタの状態を最適化コードの状態に組み替え、
ループの続きを最適化コードで走らせる。脱最適化の状態移し替えを逆向きに使う技術だと言える。
JVM や各種 JIT で、ウォームアップ中の長いループを救う標準装備になっている。

## トレーシング JIT — ホットな経路を一直線に

ここまでは「メソッド単位」で最適化する前提だった。別のアプローチが**[トレーシング](#index:トレーシング)**
（tracing）JIT だ[](#cite:gal2009)。メソッドの境界を無視し、**実際に実行された命令の一直線の並び（トレース）**を
最適化単位にする。

仕組みはこうだ。ループのヘッダなど「ホットになりやすい地点」にカウンタを置き、しきい値を超えたら
**記録モード**に入る。そこから実際に実行された命令を、分岐をどちらに進んだかも含めて一本道に記録する。
ループが一周して戻ってきたら記録を止める。こうして得た一直線のトレースには分岐合流がないので、
[定数伝播](local-optimization.md)も[型特殊化](#型特殊化-動的な演算を静的な演算に)も極めて効きやすい。

トレースは「実際に通った経路」なので、途中の分岐は**そっちに進む**という仮定になる。
だから各分岐点にガードを置き、トレードと違う方へ行きそうになったらガードが発火して
脱最適化する。トレースは「if 文を、ガード付きの一本道に開いたもの」なのだ。

```
元のループ（分岐あり）            記録したトレース（一直線＋ガード）
while x                          loop_header:
  if type(a)==Int  ──┐            guard type(a)==Int   # 外れたら脱最適化
    fast_path        │            fast_path
  else               │            guard x              # ループ継続の仮定
    slow_path  ◀─────┘            jump loop_header
```

トレーシング JIT の真価は**メタトレーシング**で発揮される。PyPy[](#cite:bolz2009)は、
ユーザプログラムではなく**インタプリタ自身**をトレースする。インタプリタが
「バイトコード命令を一つ実行するループ」を回す様子をトレースすると、その特定プログラムに
特化した実行コードが得られる——これは[前章で見たフタムラ投影](whole-program-optimization.md)を、
トレーシングで実現したものにほかならない。「インタプリタを書けば JIT がついてくる」という
Truffle の部分評価[](#cite:wurthinger2017)とは別ルートで、同じ目的地に着く。

> [!NOTE]
> トレーシング JIT（LuaJIT・PyPy・初期の JavaScript TraceMonkey[](#cite:gal2009)）と、
> メソッド JIT（HotSpot・V8・Ruby の JIT）は、長く競ってきた二大流派だ。トレーシングは
> ホットな経路を鋭く最適化できるが、経路が分岐しがちなコードでトレースが爆発しやすい。
> メソッド JIT は安定するが、メソッド境界に縛られる。どちらが優れるかは題材次第で、
> 「実行時に観測した現実を投機的に固定する」という核心は両者で共通している。

## 基本ブロックバージョニング（BBV）

最後に、比較的新しく実装が軽量な投機手法、**[基本ブロックバージョニング](#index:基本ブロックバージョニング)**
（basic block versioning; BBV）を見る[](#cite:chevalier2015)。重い型解析も別工程のプロファイラも使わず、
**基本ブロックを、入ってくる型の組み合わせごとに複製（バージョン分け）しながら、遅延生成する**。

アイデアはこうだ。あるブロックに `a` が `Integer` という型情報を持って入ってきたら、
その型を前提にした専用バージョンを**その場で**生成する。ブロックの最後でガードして型を確定し、
次のブロックへはその確定した型情報を引き継ぐ。型情報がブロックをまたいで**流れて**いくので、
一度確定した型のチェックは下流で繰り返さずに済む——型チェックが自然に消えていく。

```
       入口（a: 不明）
          │ ガードで型を確定
   ┌──────┴──────┐
a: Integer 版   a: String 版    ← 観測した型ごとにブロックを複製
   │              │
   ▼              ▼
（以降 a の型チェック不要＝型情報を引き継ぐ）
```

BBV の魅力は単純さだ。SSA も大域的データフローも要らず、「型が来たらその版を作る」だけで
[型チェック除去](dynamic-dispatch.md)が転がり込んでくる。Ruby の **YJIT**[](#cite:chevalier2021)は、
この BBV を採用した JIT で、Maxime Chevalier-Boisvert らによって CRuby に実装され、
本番の Rails アプリで実効的な高速化を達成した。「軽量な投機」という選択肢が、
言語処理系に実用的な JIT を導入する現実的な道を開いた好例だ。

「型ごとにコードを複製する」発想自体は新しくない。本章冒頭で触れた customization[](#cite:chambers1989)が
メソッド単位での複製なら、制御フローを複製して合流点での型情報の喪失を防ぐ **splitting** は、
すでに SELF で提案されていた[](#cite:chambers1990)。BBV の新しさは技法そのものより設計判断にある——
事前に解析して分割するのではなく、実行時に来た型の版だけを遅延生成する。古いアイデアを
「オンラインの投機」として組み直したことが、軽量さと実用性を両立させた鍵だ。

> [!TIP]
> トレーシング・メソッド JIT・BBV は、最適化の**単位**（一直線のトレース／メソッド／基本ブロックの版）が
> 違うだけで、設計の骨格は同じ——「型を観測し、仮定して特殊化し、ガードで守り、外れたら脱最適化する」。
> 新しい JIT に出会ったら、まずこの 4 点（単位・観測・ガード・脱出）を探すと、構造がすぐ掴める。

投機の話は「どう速いコードを作るか」だったが、その手前に「**どう速くコードを作るか**」という別の軸もある。
現代の JIT は**階層型**（tiering）で、まず**起動の速い低品質コンパイラ**で動かし始め、ホットになった部分だけを
重い最適化コンパイラにかける（V8 なら Ignition→Sparkplug→Maglev→TurboFan、WebAssembly なら
Liftoff→最適化層）。この「速いベースライン層」をどう実装するかが効く。
**copy-and-patch**[](#cite:xu2021copyandpatch)は、あらかじめコンパイルしておいた機械語の断片（ステンシル）を
穴埋めしながら継ぎ合わせるだけでコードを生成し、LLVM `-O0` の二桁（約100倍）速くコンパイルする——
CPython 3.13 の JIT もこの技法を採る。WebAssembly では、書き換えすら行わず標準バイトリストを
その場で実行する高速インタプリタ[](#cite:titzer2022inplace)が、ベースライン層のさらに下を埋めている
（「そもそもベースライン層に何を期待すべきか」という設計の問いそのものも論じられている[](#cite:titzer2024baseline)）。

## まとめ

- 投機的最適化は、観測した型を「取り消せる賭け」として仮定し、コードを特殊化する。静的最適化にはできない大胆さを、安全網と引き換えに得る。
- ガードが仮定を安く確認し、外れたら脱最適化[](#cite:holzle1992)でインタプリタへ安全に戻る。状態の再構築が技術的核心。OSR はその逆で、実行中の関数を最適化コードへ昇格させる。
- トレーシング JIT[](#cite:gal2009)はホットな経路を一直線に最適化し、メタトレーシング[](#cite:bolz2009)はインタプリタ自身をトレースしてフタムラ投影を実現する。
- BBV[](#cite:chevalier2015)は基本ブロックを型ごとに遅延複製する軽量な投機で、Ruby の YJIT[](#cite:chevalier2021)が採用した。

ここまでで、動的言語を速くする主要技法を一望した。第 V 部からは視点を機械寄りに移し、
IR を実際の機械語に翻訳する過程そのものに潜む最適化——命令選択・レジスタ割り当て・スケジューリングへ進む。
