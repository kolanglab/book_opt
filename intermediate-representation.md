# 中間表現

最適化器はソースコードを直接いじらない。`a * 2 + a * 2` という文字列を見て
「同じ式が二つある」と気づくのは、人間には簡単でも、機械には面倒だ。
そこで処理系は、まずプログラムを最適化に都合のよい**中間表現**
（[IR](#index:中間表現); intermediate representation）に翻訳してから最適化する。

この章では、本書全体で使う IR を Ruby で組み立てる。
具体的には、抽象構文木（AST）から**三番地コード**へ、それを**基本ブロック**に切り分け、
**制御フローグラフ**（CFG）を作り、最後に近代最適化の主役である **SSA 形式**まで到達する。
ここで作る道具立ては、以降のほぼすべての章で再利用される。

## なぜ AST のままではダメなのか

フロントエンドが最初に作るのは**抽象構文木**（[AST](#index:抽象構文木); abstract syntax tree）だ。
`x = a * 2 + 1` は次のような木になる。

```
        (=)
       /   \
      x    (+)
          /   \
        (*)    1
       /   \
      a     2
```

AST は構文をそのまま映すので、構文解析の出口としては自然だ。実際、
[木をたどって計算する](#index:木巡回インタプリタ)素朴なインタプリタは AST 上で書ける。
だが最適化には向かない。理由は二つある。

第一に、式が入れ子で、計算の途中結果に名前がない。
「`a * 2` の値」を指す方法がないので、「同じ計算が二度ある」を検出するのが難しい。
第二に、制御フローが木の形に隠れている。`if`/`while` が入れ子の節点になっていて、
「この文の次にどの文が実行されうるか」がグラフとして見えない。

最適化が欲しいのは、**計算が一つずつ名前付きで並び、制御の流れがグラフで見える**表現だ。
それが三番地コードと CFG である。

## 三番地コード

[三番地コード](#index:三番地コード)（[three-address code](#index:three-address code); TAC）とは、
各命令がたかだか「`結果 = オペランド1 演算 オペランド2`」という、
3 つの番地（オペランド）しか持たない形式の IR である[](#cite:aho2006)。
入れ子の式は、コンパイラが生成する一時変数（temporary; `t0, t1, ...`）に分解される。

先ほどの `x = a * 2 + 1` は、次の 2 命令になる。

```
t0 = a * 2
x  = t0 + 1
```

Ruby で命令を表す小さなクラスを作ろう。本書ではこれを最小の IR として使い回す。

```ruby
# 一つの三番地コード命令。op はシンボル、dst は代入先、args はオペランド配列。
Instr = Struct.new(:op, :dst, :args) do
  def to_s
    case op
    when :assign then "#{dst} = #{args[0]}"
    when :binop  then "#{dst} = #{args[1]} #{args[0]} #{args[2]}"
    when :goto   then "goto #{args[0]}"
    when :branch then "if #{args[0]} goto #{args[1]} else #{args[2]}"
    when :ret    then "ret #{args[0]}"
    when :phi    then "#{dst} = phi(#{args.join(', ')})"  # 後述
    else "#{dst} = #{op}(#{args.join(', ')})"
    end
  end
end
```

AST から三番地コードへの変換は、木を帰りがけ（後行順）にたどり、
部分式ごとに一時変数を割り当てていくだけだ。これを**式の平坦化**（flattening）と呼ぶ。

```ruby
class Flattener
  def initialize = (@code = []; @counter = 0)
  def fresh = "t#{@counter += 1}"            # 新しい一時変数名

  # AST ノードを平坦化し、「その値を保持する変数名」を返す
  def gen(node)
    case node
    in [:num, n]          then n.to_s
    in [:var, name]       then name
    in [:binop, op, l, r]
      lt, rt = gen(l), gen(r)                              # 左右を先に平坦化
      t = fresh                                            # その後で一時変数を採番
      @code << Instr.new(:binop, t, [op, lt, rt])
      t
    in [:assign, name, e]
      @code << Instr.new(:assign, name, [gen(e)]); name
    end
  end

  def run(ast) = (gen(ast); @code)
end

ast = [:assign, "x", [:binop, :+, [:binop, :*, [:var, "a"], [:num, 2]], [:num, 1]]]
Flattener.new.run(ast).each { puts _1 }
# t1 = a * 2
# t2 = t1 + 1
# x = t2
```

平坦化された途中結果に名前が付いたことで、
「同じ右辺を持つ命令」を探すだけで共通部分式が見つかるようになった。
これが [local-optimization](local-optimization.md) の出発点になる。

## 基本ブロックと制御フローグラフ

次に制御の流れを表現する。命令列を、分岐の切れ目で区切ったかたまりが**基本ブロック**だ。
正確には、**リーダ**（leader; ブロックの先頭になりうる命令）を次の規則で決める[](#cite:aho2006)。

1. 最初の命令はリーダ。
2. 分岐命令（`goto`/`branch`）の飛び先はリーダ。
3. 分岐命令の直後の命令はリーダ。

各リーダから次のリーダの手前までが一つの基本ブロックになる。
ブロックを節点、分岐を辺とした有向グラフが**制御フローグラフ**（[CFG](#index:制御フローグラフ); control flow graph）だ。

例として、絶対値を計算する関数を考える。

```
       B0: if a < 0 goto B1 else B2
      /                          \
B1: r = 0 - a                 B2: r = a
      \                          /
       B3: ret r
```

`B0` から `B1` と `B2` に分岐し、両者が `B3` に合流する。
この「合流」が後で SSA の鍵になるので覚えておこう。Ruby で表すなら、
ブロックは「命令の配列」、CFG は「ブロック ID から後続ブロック ID の集合への対応」だ。

```ruby
BasicBlock = Struct.new(:id, :instrs, :succs, :preds) do
  def initialize(id) = super(id, [], [], [])   # 命令列・後続・先行
end

class CFG
  attr_reader :blocks
  def initialize = (@blocks = {})
  def block(id) = (@blocks[id] ||= BasicBlock.new(id))
  def edge(from, to)                            # from -> to の辺を張る
    block(from).succs << to
    block(to).preds  << from
  end
end
```

CFG ができると、最適化の多くが「グラフ上の問い」に翻訳できる。
「このブロックに来る前に必ず通るブロックはどれか」（[支配関係](#index:支配関係)、後述）、
「この変数の値はどのブロックから流れてくるか」。
こうした問いに答えるのが、次章の[データフロー解析](dataflow-analysis.md)だ。

> [!NOTE]
> **先行ブロック**（predecessor）と**後続ブロック**（successor）という語は本書で頻出する。
> ブロック B から辺が出ている先が後続、B に辺が入ってくる元が先行だ。
> 上の例で `B3` の先行は `{B1, B2}`、`B0` の後続は `{B1, B2}` である。

## 支配関係

CFG 上で「ブロック B に到達するには、必ずブロック A を通らねばならない」とき、
**A は B を支配する**（A dominates B）という[](#index:支配関係)。
入口ブロックはすべてのブロックを支配する。各ブロック B にとって、
B 自身を除いて B に最も近い支配ブロックを**直接支配者**（immediate dominator, idom）と呼ぶ。
idom を辺で結ぶと**支配木**（dominator tree）ができる。

支配関係は、最適化の安全性を語る共通言語だ。たとえば「定義 `x = ...` が使用 `... = x` を
支配する」なら、その使用に届く値はその定義（か、より後の再定義）に限られる。
[ループ不変式の移動](loop-optimization.md)も「移動先がループ内のすべての使用を支配するか」で正しさが決まる。

支配者の素朴な計算は、データフロー解析の典型例なので次章で式とともに扱う。
実用的には Lengauer–Tarjan のほぼ線形時間アルゴリズム[](#cite:lengauer1979)や、
反復だが実装が単純で実測も速い Cooper–Harvey–Kennedy 法[](#cite:cooper2001)が使われる。
ここでは「支配＝必ず通る、という関係であり、支配木という形で持てる」とだけ押さえておこう。

## SSA 形式

近代的な最適化器の標準 IR が**SSA 形式**（[static single assignment form](#index:SSA形式); 静的単一代入形式）だ
[](#cite:cytron1991)。規則はひとつ、**各変数への代入はプログラム中ちょうど一箇所**だ。
同じ変数に二度代入したくなったら、番号を振って別変数にする。

```
普通の IR            SSA 形式
x = 1            →   x1 = 1
x = x + 1        →   x2 = x1 + 1
y = x * 2        →   y1 = x2 * 2
```

なぜこれが嬉しいのか。SSA では**変数名がそのまま「その値を生んだ唯一の定義」を指す**。
「`x2` の値はどこから来たか」を探す必要がない。定義は一つしかないからだ。
この性質（[def-use 関係](#index:def-use関係)が自明になること）のおかげで、
定数伝播・値番号付け・不要コード除去といった最適化が劇的に単純化する（[ssa-optimization](ssa-optimization.md)）。

問題は合流点だ。先ほどの絶対値の例で `B3` に来る `r` は、
`B1` を通れば `r = 0 - a` の値、`B2` を通れば `r = a` の値であり、
来た経路によって違う。SSA では一つの変数に一つの定義しか許さないので、これは困る。

そこで導入されるのが [**φ（ファイ）関数**](#index:φ関数)（phi function）だ。
合流ブロックの先頭に「先行ブロックごとにどの値を選ぶか」を表す擬似命令を置く。

```
B1: r1 = 0 - a1
B2: r2 = a1
B3: r3 = phi(r1 from B1, r2 from B2)   # B1 から来たら r1、B2 から来たら r2
    ret r3
```

φ 関数は「先行ブロックの数だけ引数を持ち、実行時に来た経路に応じてどれか一つを選ぶ」
という意味の、概念上の命令だ。実機にこんな命令はないので、SSA を抜けるとき
（[レジスタ割り当て](register-allocation.md)の前）に、各先行ブロックの末尾へのコピー
（`r3 = r1` など）に変換して消す。これを **SSA からの脱出**（SSA destruction）と呼ぶ。

φ をどのブロックに置けばよいかは、闇雲に全合流点へ置くと無駄が多い。
Cytron らは、変数の定義のある場所の**支配辺境**（dominance frontier）に置けば、
過不足ない最小限の φ で済むことを示した[](#cite:cytron1991)。

支配辺境とは、あるブロック X から見て、「**X を必ず通ってくる経路と、通らずに来る経路とが初めて出会う点**」のことだ。
小さな if/else で見よう。

```
        B0  (x = 0)
       /  \
     B1    B2         B1 では x = 1 と書き換える
       \  /
        B3  (… = x)   ← B1 を通った x=1 と、通らなかった x=0 がここで初めて合流する
```

`x` は B0 と B1 で定義される。B1 で書いた `x=1` は B1 を通った経路だけのものだが、B3 は B2 経由
（B1 を通らない）でも到達できる。だから B3 が「B1 の影響が及ぶ範囲の**ちょうど外側**」＝B1 の支配辺境であり、
ここに φ を置いて「B1 から来たか（`x=1`）、そうでないか（`x=0`）」を選ばせればよい。
詳しい構成法は [ssa-optimization](ssa-optimization.md) で扱う。SSA への変換は「番号付け＋φ 挿入」の二段で、
支配関係に支えられている。いまはそう掴めば十分だ。

> [!TIP]
> SSA は今や LLVM IR[](#cite:lattner2004)、Java HotSpot、V8、各種 JIT の共通基盤だ。
> さらに進んだ「[sea of nodes](#index:sea of nodes)」[](#cite:click1995ir) という、
> 制御フローとデータフローを一枚のグラフに融合した IR もあり、HotSpot サーバコンパイラや
> GraalVM が採用している。本書では基本ブロック＋φ の古典的 SSA を主に使うが、
> [ssa-optimization](ssa-optimization.md) と [equality saturation](equality-saturation.md) で sea of nodes 系の発想にも触れる。
>
> IR 設計に「正解」はなく、今も揺れている。命令を CFG に縛らず浮かせる sea of nodes は
> スケジューリングの自由度を生む一方、解析やデバッグを難しくする。実際 2025 年、V8 は
> sea of nodes を捨てて CFG ベースの IR（Turboshaft/Maglev）へ回帰すると表明した[](#cite:v8t2025leavingson)。
> 「純粋なノードだけが浮く SoN は、結局 CFG とそう変わらない」というのが現場の総括だ。
> 逆に、データ依存を主役に据えてループや分岐を領域として畳む RVSDG[](#cite:reissmann2020rvsdg)のように、
> 制御フローをさらに脱構築する方向の研究もある（PDG[](#cite:ferrante1987pdg)の系譜で、
> こうした需要依存グラフからでも元の制御フローを完全に復元できることが示されている[](#cite:bahmann2015reconstruct)）。
> 「制御フローをどこまで陽に持つか」は、いまだ生きた設計問題なのだ。

## 本書で使う IR の地図

この章で、ソースから最適化用 IR までの道のりを一通り組み立てた。整理しておく。

- **AST** は構文に忠実だが、途中結果に名前がなく制御がグラフで見えないので最適化に不向き。
- **三番地コード**で計算を平坦に並べ、途中結果に名前を与える。共通部分式が見えるようになる。
- **基本ブロック**に切り分け、**CFG** で制御の流れをグラフにする。最適化が「グラフの問い」になる。
- **支配関係**で「必ず通る」を捉え、最適化の安全性を語る共通言語にする。
- **SSA 形式**で各変数を単一定義にし、φ 関数で合流を扱う。def-use が自明になり最適化が単純化する。

次章では、この CFG の上で「どの地点で何が成り立つか」を機械的に計算する
**データフロー解析**を学ぶ。最適化の安全性は、ほぼすべてここから供給される。
