# データフロー解析 — 何が成り立つかを機械的に知る

前章で、最適化の正しさは「ある前提が成り立つこと」に依存すると述べた。
「この式の値は定数か」「この変数はこの先で使われるか」「この代入は到達するか」——
こうした問いに、CFG 全体を見渡して**機械的に**答える枠組みが
[データフロー解析](#index:データフロー解析)（data-flow analysis）だ[](#cite:kildall1973)。

データフロー解析は最適化の屋台骨である。本書で扱うほとんどの最適化は、
「まずデータフロー解析で安全に変換できる場所を見つけ、次に変換する」という二段構えになっている。
この章では、生存変数解析を例に解析の機械を組み立て、それを支える理論——
格子と不動点[](#cite:kildall1973)、その一般化である抽象解釈[](#cite:cousot1977)——まで一気に登る。

## 動かす例 — 生存変数解析

[生存変数解析](#index:生存変数解析)（live variable analysis）を題材にしよう。
ある地点で変数 `v` が**生存している**（live）とは、「その地点から先で、`v` が
再代入される前に少なくとも一度**読まれる**経路がある」こと。
逆に、どの経路でも二度と読まれないなら `v` は**死んでいる**（dead）。

これが分かると何が嬉しいか。死んでいる変数への代入は[不要コード除去](dead-code-elimination.md)で消せるし、
生きている変数の集合が分かれば[レジスタ割り当て](register-allocation.md)で「同時に生きている変数の数」
＝必要なレジスタ数の見積もりに使える。生存変数は最も使い出のある解析の一つだ。

各命令について、まず局所的な二つの集合を定義する。

- **use**: その命令が（再代入の前に）**読む**変数の集合。
- **def**: その命令が**書く**変数の集合。

`x = a + b` なら `use = {a, b}`、`def = {x}`。これを命令単位ではなく
基本ブロック単位でまとめたものを、それぞれ `USE[B]`、`DEF[B]` と書く
（`DEF[B]` は「B 内で、読まれる前に書かれる変数」）。

## データフロー方程式 — 流れを式にする

解析は、各ブロック B の入口・出口で生存している変数集合 `IN[B]`、`OUT[B]` を求める。
生存情報は**実行と逆向き**に流れる（「この先で使うか」を問うので未来から過去へ）。
これを**後方解析**（backward analysis）という。方程式はこうなる。

$$
\mathit{OUT}[B] = \bigcup_{S \in \mathit{succ}(B)} \mathit{IN}[S]
$$

$$
\mathit{IN}[B] = \mathit{USE}[B] \cup (\mathit{OUT}[B] \setminus \mathit{DEF}[B])
$$

言葉にすると——ブロック B の出口で生きている変数は、**後続のどれかの入口で生きている**変数すべて
（合流; meet）。B の入口で生きている変数は、B 内で使う変数（`USE`）に、
出口で生きていて B 内で上書きされなかった変数を足したもの（**伝達関数**; transfer function）。

この `IN`/`OUT` は、最初は空集合から始めて、方程式が**もう変化しなくなるまで**
（[不動点](#index:不動点); fixed point に達するまで）繰り返し代入すれば求まる。
集合は単調に増え、変数の総数で頭打ちなので、必ず有限回で止まる。

```ruby
# blocks: {id => {use:, def:, succs:}} を与え、各ブロックの IN/OUT を返す
def live_variables(blocks)
  inn = Hash.new { |h, k| h[k] = Set.new }
  out = Hash.new { |h, k| h[k] = Set.new }
  loop do
    changed = false
    blocks.each_key do |b|
      new_out = blocks[b][:succs].reduce(Set.new) { |acc, s| acc | inn[s] }   # 合流
      new_in  = blocks[b][:use] | (new_out - blocks[b][:def])                 # 伝達
      if new_out != out[b] || new_in != inn[b]
        out[b], inn[b] = new_out, new_in
        changed = true
      end
    end
    break unless changed   # どのブロックも動かなくなったら不動点
  end
  [inn, out]
end
```

ここで二つの設計判断が現れている。**逆向き**（後続の `IN` を集める）に流すこと、
そして合流を**和集合**（`|`）で取ること。これらは「生存変数」という問題に固有の選択だ。
別の解析では向きや合流演算が変わる。次の表に主な解析を並べる。

| 解析 | 向き | 合流 | 何が分かるか | 使い道 |
|------|------|------|------------|--------|
| 生存変数 | 後方 | ∪ | この先で使われるか | 不要コード除去・レジスタ割当 |
| 到達定義 | 前方 | ∪ | どの代入が届くか | 定数伝播・コピー伝播 |
| 利用可能式 | 前方 | ∩ | 既に計算済みの式 | 共通部分式除去 |
| 非常に忙しい式 | 後方 | ∩ | 必ず後で使う式 | コードの巻き上げ |

> [!NOTE]
> **前方／後方**は情報が流れる向き、**∪／∩**は合流のしかただ。
> ∪ を取る解析は「ある経路で成り立てばよい（might）」を、∩ を取る解析は
> 「すべての経路で成り立つ（must）」を意味する。
> 生存は「ある経路で使われれば生存」なので ∪、利用可能式は「すべての経路で計算済みでないと使い回せない」ので ∩。
> この四象限を押さえると、新しい解析に出会っても向きと合流を当てるだけで骨格が分かる。

## なぜ正しく止まるのか — 格子と単調性

上の反復が「必ず止まり、しかも一意の答えに収束する」のは偶然ではない。
背後に**[束（格子）](#index:束)**（lattice）の理論がある[](#cite:kildall1973)。

解析が扱う値の集合（ここでは「変数の集合」全体）に、順序 ⊑ を入れる。
生存変数なら「集合の包含 ⊆」を順序とし、合流 ⊔ を「和集合 ∪」とする。
この構造が次を満たすとき、反復解析はうまくいく。

- **有限の高さ**: 順序の鎖（だんだん大きくなる列）はいつか頭打ちになる。
  → だから反復は有限回で止まる。
- **伝達関数が単調**: 入力が（順序の意味で）大きくなれば出力も大きくなる。
  $x \sqsubseteq y \Rightarrow f(x) \sqsubseteq f(y)$。
  → だから反復で値が後戻りせず、一方向に収束する。

この二条件があれば、初期値（最小元 ⊥）から反復して得られる不動点は
**最小不動点**であり、開始順序によらず一意に定まる。Kildall はこの枠組みで、
定数伝播・共通部分式除去などを「同じ反復アルゴリズムの異なるインスタンス」として
統一的に扱えることを示した[](#cite:kildall1973)。データフロー解析が「枠組み」と呼ばれるゆえんだ。

実装上は、ワークリスト法で無駄な再計算を省くのが定石だ。
「変化したブロックの隣だけを再訪する」ことで、全ブロック総なめの反復を避ける。

```ruby
def live_variables_worklist(blocks)
  inn = Hash.new { |h, k| h[k] = Set.new }
  out = Hash.new { |h, k| h[k] = Set.new }
  worklist = blocks.keys.to_a            # 最初は全ブロックを処理対象に
  until worklist.empty?
    b = worklist.pop
    new_out = blocks[b][:succs].reduce(Set.new) { |a, s| a | inn[s] }
    new_in  = blocks[b][:use] | (new_out - blocks[b][:def])
    out[b] = new_out
    if new_in != inn[b]                  # 入口が変わったら…
      inn[b] = new_in
      worklist |= blocks[b][:preds]      # …先行ブロックを再処理キューへ
    end
  end
  [inn, out]
end
```

後方解析なので、入口 `IN[b]` が変わったら**先行**（preds）を再訪する点に注意。
前方解析なら逆に、出口が変わったら後続を再訪することになる。

## 解析の精度 — パス感度と限界

データフロー解析は CFG の**すべての経路**を一様に考える。
これは安全だが、ときに悲観的すぎる。次の例を見よう。

```ruby
if cond
  x = 1
else
  x = 2
end
# ここで x は「1 か 2」。素朴な定数伝播は「定数ではない」と諦める
```

二つの経路から `1` と `2` が合流するので、`x` を単一の定数とは言えない。
これは正しい判断だが、「実際にはどちらの経路を通るか」までは古典的な解析は追わない。
経路ごとに別々の事実を持つ解析を**パス感度のある**（path-sensitive）解析と呼び、
精度は上がるが経路数の爆発と引き換えになる。動的言語ではこの限界を、
実行時に観測した「実際に通った経路」で補う——それが第 IV 部の[投機的最適化](speculative-optimization.md)の発想だ。

さらに、ポインタや参照が絡むと「`p` と `q` は同じ場所を指すか」（別名）が分からず、
`*p = 1` が `q` の値を変えるかどうか判断できない。これを解くのが
[手続き間解析の章](interprocedural-analysis.md)で扱う別名解析[](#cite:steensgaard1996)だ。
古典的データフロー解析は、別名や手続き呼び出しの前では「最悪を仮定して」安全側に倒す。

## 抽象解釈 — 解析を設計する理論

ここまでは個別の解析を作ってきたが、「正しい解析をどう設計し、その正しさをどう保証するか」
を体系化したのが Cousot 夫妻の**[抽象解釈](#index:抽象解釈)**（abstract interpretation）だ[](#cite:cousot1977)。

中心アイデアは、プログラムを**実際の値ではなく、抽象的な値で実行（評価）する**こと。
たとえば整数を「具体的な数」ではなく「符号（負・零・正・不明）」で追えば、
`(負) * (負) = (正)` のように、計算を抽象世界で再現できる。

```ruby
# 符号領域での乗算。具体的な値を知らなくても符号は伝播できる
def mul_sign(a, b)
  return :zero if a == :zero || b == :zero
  return :unknown if a == :unknown || b == :unknown
  (a == b) ? :pos : :neg     # 同符号なら正、異符号なら負
end
mul_sign(:neg, :neg)  # => :pos
```

この「符号」のような抽象領域（abstract domain）と、具体と抽象を結ぶ対応
（[ガロア接続](#index:ガロア接続); Galois connection）をきちんと定めると、
抽象世界での実行結果が**つねに安全な近似**であることが理論的に保証される[](#cite:cousot1977)。
符号領域、区間領域（`x ∈ [0, 10]`）、定数領域など、領域を取り替えるだけで
別々の解析が「同じ枠組みの異なる設定」として導かれる。先ほどの定数伝播も、
「定数か `⊤`（不明）か」という抽象領域での抽象解釈だと見なせる。

無限に高い領域（区間など）では反復が止まらないことがあり、
わざと精度を落として収束させる**ワイドニング**（widening）という操作で対処する。
抽象解釈は本書のスコープを超える深い理論だが、「**安全な近似を体系的に作る方法**」
という一文で、データフロー解析の正しさの源泉を覚えておこう。
