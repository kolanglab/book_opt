# 不要コード除去

これまでの最適化の多くは、式をより速い形へ書き換えるものだった。だが速くする手はそれだけではない。
実行しても結果が使われない計算を、まるごと消すのも有効だ。しかも書き換え系の最適化は、しばしば
この「消せる計算」を生み出す。前章のコピー伝播は `a = b` を不要にし、定数伝播は `x = 3` を使われない代入に変えた。
こうした「結果がどこにも使われない計算」を取り除くのが
[不要コード除去](#index:不要コード除去)（dead code elimination; DCE）だ。

DCE は単純そうで奥がある。「使われていない」を正しく判定するには
[生存変数解析](dataflow-analysis.md)が要るし、副作用のある計算を消すと意味が壊れる。
さらに「到達しないコード」（dead code とは別物）の除去や、SSA を使った効率的な版もある。
この章でそれらを整理し、Ruby で実装する。

## 二種類の「死んだコード」

まず用語を区別しておく。日本語ではどちらも「デッドコード」と呼ばれがちだが、別物だ
[](#cite:muchnick1997)。

- **[不要コード](#index:不要コード)（dead / useless code）**: 実行はされるが、
  その結果が観測可能な振る舞いに一切影響しない計算。例：誰も読まない変数への代入。
- **[到達不能コード](#index:到達不能コード)（unreachable / dead code）**: どの実行経路からも
  到達しない命令。例：`return` の後ろの文、つねに偽の `if` の中身。

到達不能コードの除去は CFG の到達可能性（入口から辿れるか）で判定できる、純粋な制御フローの問題だ。
一方の不要コード除去は「結果が使われるか」というデータフローの問題で、生存変数解析を使う。

なお、章冒頭で「消すと速くなる」と述べたのは、実行はされるが結果が使われない不要コードの話だ。
到達不能コードはそもそも実行されないので、消しても実行パスは直接は速くならない。その利益は主に間接的で、
(1) 後続の最適化を呼び込むこと（次段落のカスケード）、(2) コードサイズの減少によるコンパイル時間・
命令キャッシュ・JIT のコードキャッシュ（有限資源）の改善にある。
本章は主に後者を扱うが、両者は連鎖する。到達不能ブロックを消すと、そこでしか使われていなかった
計算が不要になり、不要コードを消すと、そこへの分岐だけだったブロックが到達不能になる。

## 生存変数解析にもとづく DCE

基本のアルゴリズムはこうだ。[生存変数解析](dataflow-analysis.md)で各地点の生存集合を求め、
**結果（dst）がその命令の直後で生存していない代入命令を削除する**。
ただし副作用のある命令は、結果が使われなくても消すわけにはいかない。

```ruby
# code: 命令配列（単一ブロックを仮定）。out_live: 末尾で生存している変数の集合。
def dead_code_elimination(code, out_live = Set.new)
  loop do
    live = out_live.dup
    keep = []
    code.reverse_each do |i|           # 生存は後方解析なので末尾から前へ
      dead = i.dst && !live.include?(i.dst) && !side_effect?(i)
      next if dead                     # 結果が死んでいて副作用もない → 削除
      keep << i
      live.delete(i.dst) if i.dst      # この命令が dst を定義（dst はこの上では死ぬ）
      live.merge(uses(i))              # オペランドはこの上で生存
    end
    new_code = keep.reverse
    return new_code if new_code.size == code.size   # 何も消えなければ終了
    code = new_code                    # 消えたら、新たな死を生んだかもしれないので再走
  end
end

def side_effect?(i) = %i[call store ret branch goto print].include?(i.op)
def uses(i) = i.args.select { |a| a.is_a?(String) && a !~ /\A-?\d+\z/ }.to_set
```

二つの設計判断に注目してほしい。第一に後方に走査する点（生存は未来から流れてくる）。
第二に、削除のたびにもう一度回す点。`c = a + b` を消すと `a` と `b` の使用が一つ減り、
それらが他で使われていなければ今度は `a`/`b` の定義が不要になる。
DCE は連鎖反応するので、不動点まで繰り返す。

```ruby
src = [Instr.new(:binop, "a", [:+, "x", "y"]),
       Instr.new(:binop, "b", [:*, "a", "2"]),   # b は誰も使わない
       Instr.new(:assign, "r", ["x"]),
       Instr.new(:ret,    nil, ["r"])]
dead_code_elimination(src).each { puts _1 }
# r = x
# ret r
#   → a も b も結果が使われないので、連鎖的に両方消える
```

> [!WARNING]
> `side_effect?` の網羅が DCE の生命線だ。メソッド呼び出し（`call`）は、戻り値を捨てても
> 内部で I/O やインスタンス変数の書き換えをするかもしれないので、原則消せない。
> 「この呼び出しは純粋（副作用なし）だ」と分かって初めて消せる。その判定は
> [手続き間解析](interprocedural-analysis.md)の領分だ。安全側に倒すなら「呼び出しは消さない」が正解。
> 副作用を見落として消すのは、最適化バグの典型例である。

## SSA を使った効率的な DCE

[SSA 形式](intermediate-representation.md)を使うと、DCE はもっと速く・単純になる。
SSA では各変数が単一定義なので、わざわざ反復データフローを回さなくても、
**「使われている定義」をワークリストで辿るだけ**で済む[](#cite:cytron1991)[](#cite:muchnick1997)。

考え方は到達可能性のマーキングだ。まず「観測可能な命令」（`ret`、`store`、`call`、副作用持ち）を
生きていると種付けする。次に、生きている命令が使うオペランドの定義を生きていると印を付け、
それを波及させる。最後に印の付かなかった命令を一掃する。

```ruby
def ssa_dce(instrs)
  def_of = instrs.to_h { |i| [i.dst, i] }     # SSA: 変数 → その唯一の定義
  live = Set.new
  worklist = instrs.select { |i| side_effect?(i) }   # 種：副作用のある命令

  until worklist.empty?
    i = worklist.pop
    next if live.include?(i)
    live << i
    uses(i).each do |u|                       # 使っている各変数の定義を生かす
      d = def_of[u]
      worklist << d if d && !live.include?(d)
    end
  end
  instrs.select { |i| live.include?(i) }      # 印のない命令を削除
end
```

これは「ゴミ集めのマーク＆スイープ」とそっくりだ。
根（副作用命令）から到達できる定義だけを残す。SSA のおかげで def-use が一意なので、
反復不要の一回のグラフ探索で答えが出る。SSA がいかに最適化を単純化するかの好例だ。

## φ と分岐まで消す攻めの不要コード除去

素朴な DCE には弱点がある。ループのために自分自身を使い合うコードを消せないのだ。

```ruby
# i は使われていないのに、ループのたびに自分を更新するので「使われている」ように見える
i = 0
while cond
  i = i + 1     # i の唯一の使用は、この i+1 自身。本当は無駄
end
```

`i` の使用は `i + 1` だけ、その `i + 1` の使用も次の周回の `i + 1` だけだ。
自己参照のループで、素朴なマーキングは全部「生きている」と判定してしまう。
これを消すには発想を逆にする。最初は全部死んでいると仮定し、本当に必要なものだけを生かす
**[攻めの不要コード除去](#index:攻めの不要コード除去)**（aggressive DCE）だ[](#cite:cytron1991)。

種にするのは副作用命令と、それらが制御依存する分岐だけ。
上のループの `i = i + 1` はどの副作用命令にも（直接にも制御依存でも）必要とされないので、
「生きている」根に辿り着けず、まとめて消える。攻めの DCE は「使われているから残す」ではなく
「観測に貢献すると証明できるものだけ残す」という、より強い立場を取る。
LLVM の `adce` パスなどが採用している。

ここで**制御依存**（control dependence）とは、「ある分岐の結果によって、
その命令が実行されるか否かが決まる」関係のこと。攻めの DCE が分岐まで消せるのは、
この関係を使って「この分岐は、生きている命令の実行可否に関与するか」を判定するからだ。
制御依存は [intermediate-representation.md](intermediate-representation.md) で触れた支配の双対概念
（逆向きの支配＝後支配）から計算でき、Cytron らの SSA 構築論文で詳しく扱われている[](#cite:cytron1991)。

> [!NOTE]
> 「最初は全部死んでいると仮定し、必要なものだけ生かす」という攻めの姿勢は、本書で繰り返し現れる。
> 次章の SCCP[](#cite:wegman1991) も「最初は全ブロック到達不能・全変数 ⊤ と仮定」して始める。
> 楽観的に出発して証明できたものだけ採用するこの構えは、悲観的な保守解析より強い結果を出せることがある。

## まとめ

- 「不要コード（結果が使われない）」と「到達不能コード（実行されえない）」は別物。前者はデータフロー、後者は制御フローの問題。
- 基本の DCE は生存変数解析＋削除の不動点反復。副作用のある命令は結果が使われなくても消せない。
- SSA を使うと、副作用命令を種にした一回のマーキングで DCE が済み、反復データフローは要らない。
- 攻めの DCE は「全部死んでいる」と仮定して必要なものだけ生かし、自己参照ループや無駄な分岐まで消す。

次章は、この SSA の上で定数伝播・値番号付け・DCE が**合流点を越えて**働く大域版へ進む。
SSA が「局所最適化を関数全体に持ち上げる装置」であることを、SCCP と GVN で確かめよう。
