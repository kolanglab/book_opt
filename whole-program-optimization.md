# 全プログラム最適化 — 視野の広さが質に変わる

最適化の視野を限界まで広げると、**プログラム全体**が一度に見える。
「このメソッドは結局この一つの実装しか呼ばれない」「このクラスは一度も継承されない」
「この設定値はビルド全体で定数だ」——個々の関数を見ていては分からないこうした事実は、
全体を見渡して初めて確定する。[全プログラム最適化](#index:全プログラム最適化)
（whole program optimization）は、この大局観でしかできない変換を行う。

この章では、全体視野の三つの果実——クラス階層解析による**脱仮想化**[](#cite:dean1995)、
翻訳単位を越える**リンク時最適化（LTO）**、そして「インタプリタとプログラムを融合して
専用コードを得る」**部分評価**[](#cite:futamura1999)——を扱う。視野の広さがいかに最適化の質に化けるかを見る。

## 脱仮想化 — 動的な呼び出しを直接呼び出しに

オブジェクト指向言語の `obj.method()` は**仮想呼び出し**（virtual call）だ。
`obj` の実行時の型によって呼び先が変わるので、実行時にメソッド表を引く間接呼び出しになる。
これは遅いだけでなく、呼び先が分からないために[インライン展開](inlining.md)を阻む——
最適化の母を呼べないのが最大の損失だ。

**[脱仮想化](#index:脱仮想化)**（devirtualization）は、仮想呼び出しを**直接呼び出し**に——
できればインラインに——変える最適化だ。鍵は「この呼び出しの呼び先は実は一つだけ」と証明すること。
静的にこれを行う代表が**クラス階層解析（CHA）**[](#cite:dean1995)である。

CHA は、プログラム全体のクラス階層を見て、`obj` の静的型 T が実行時に取りうるクラス——
**T とそのすべてのサブクラス**（T の「コーン」）——を列挙する。そのコーンに属するクラスが
`method` を呼んだとき、**全員が同一の実装に解決される**なら、呼び先はその一つに確定する、と判断する。
「オーバーライドしているのが一つだけ」では足りないことに注意——未オーバーライドのサブクラスや、
基底クラス自身のインスタンスは基底の実装へ行くので、コーン全体が一つの実装に**収束する**ことが条件だ。

```ruby
class Shape;          def area = 0;                  end
class Circle < Shape; def area = Math::PI * @r ** 2; end
class Ring   < Circle;                                end  # area を上書きしない → Circle#area を継承

def total(circles)
  # circles の要素の静的型が Circle なら、そのコーンは {Circle, Ring}。
  # 両者とも area を Circle#area に解決するので呼び先は一つに確定（CHA）。
  circles.sum { |c| c.area }   # → 直接呼び出し → インライン展開できる
end
# 一方 static type が Shape だと、コーンは {Shape, Circle, Ring} で
# Shape#area と Circle#area の二つに割れるため、CHA では確定できない。
```

CHA をさらに絞るのが**Rapid Type Analysis（RTA）**[](#cite:bacon1996)だ。CHA は「定義されている
サブクラス」を見るが、RTA は「プログラム中で実際に `new` される（インスタンス化される）クラス」だけに
限る。定義はされていても一度も生成されないクラスは呼び先候補から外せるので、CHA より精密になる。
Bacon と Sweeney は、RTA が C++ の仮想呼び出しの 7 割超を解決できたと報告した[](#cite:bacon1996)。

> [!WARNING]
> 静的な脱仮想化の大敵は**開世界仮定**（open-world assumption）だ。Java の動的クラスロードや
> Ruby の `Shape` への後付けメソッド定義のように、**実行中にクラス階層が増える**言語では、
> 「今は呼び先が一つ」でも将来覆りうる。だから動的言語・動的ロード環境では、CHA の結論を
> **投機的に**使い、「仮定が崩れたら脱最適化する」のが定石になる（[投機的最適化](speculative-optimization.md)）。
> 静的な確定と、動的な「賭けて戻せる確定」——同じ脱仮想化でも、世界の開閉で手段が変わる。

## リンク時最適化（LTO）

伝統的なコンパイルは、ソースファイル（翻訳単位）ごとに独立して最適化し、
最後にリンカがオブジェクトファイルをつなぐ。この方式だと、**ファイルをまたぐ**最適化機会——
別ファイルの小関数のインライン、ファイル越しの定数伝播、未使用関数の削除——が
すべて取り逃される。各ファイルは「他のファイルのことを知らない」からだ。

**[リンク時最適化](#index:リンク時最適化)**（link-time optimization; LTO）は、各翻訳単位を
機械語ではなく**中間表現のまま**オブジェクトファイルに格納し、リンク時にそれらを集めて
**プログラム全体を一つの IR として再最適化**する[](#cite:lattner2004)。LLVM の `-flto` がこの仕組みで、
ビットコードをリンク段で結合してから手続き間最適化を回す。

```
通常:   a.c → [最適化] → a.o ─┐
        b.c → [最適化] → b.o ─┴→ [リンク] → 実行ファイル
                                   ↑ ファイル境界を越えた最適化はできない

LTO:    a.c → a.bc(IR) ─┐
        b.c → b.bc(IR) ─┴→ [全体を結合して最適化] → [リンク] → 実行ファイル
                              ↑ ここで全体視野のインライン・DCE・定数伝播
```

LTO の典型的な利得は、(1) ライブラリ境界を越えたインライン、(2) プログラム全体で
一度も呼ばれない関数の除去（手続き間 DCE）、(3) 全体で定数と分かる値の伝播、だ。
代償はリンク時間とメモリ——全体を一度に最適化するので重い。これを和らげるのが
**ThinLTO**で、全体を一枚にせず「関数の要約（[前章](interprocedural-analysis.md)の summary）」だけを
全体で共有し、インラインに必要な関数だけを取り寄せて並列に最適化する。スケーラビリティのための妥協だ。

## プロファイル誘導最適化（PGO）

全プログラムを見渡せても、「どの経路がホットか」は静的には分からない。それを補うのが
**[プロファイル誘導最適化](#index:プロファイル誘導最適化)**（profile-guided optimization; PGO）だ。
一度プログラムを計測用にビルドして実行し、「どの分岐がどちらに転びやすいか」「どの呼び出しが
頻繁か」を記録する。その**プロファイル**を二度目のビルドに食わせ、最適化判断の根拠にする。

PGO が効く場面は広い。インライン判断（ホットな呼び出しを優先展開）、
[基本ブロックの配置](inlining.md)（ホットなブロックを連続させ命令キャッシュ効率を上げる、
コールドなコードを末尾へ追い出す）、分岐の並べ替え、レジスタ割り当ての優先度——
どれも「どこが熱いか」が分かって初めて賢く決められる。静的なヒューリスティクスが推測していたものを、
PGO は実測で置き換える。JIT が実行時プロファイルで行うことを、AOT（事前コンパイル）の世界で
二段ビルドにより実現したもの、と見ることもできる。

> [!NOTE]
> PGO の弱点は「計測時の入力が本番と違うと、最適化がずれる」こと。本番の負荷を代表する
> プロファイルを取れるかが成否を分ける。近年は、本番環境で軽量にサンプリングし続けて
> 継続的に最適化に反映する仕組み（Google の AutoFDO[](#cite:chen2016autofdo) など）もある。
> 「最適化判断の根拠を、推測から実測へ」という流れは、[機械学習による最適化](ml-and-future.md)とも地続きだ。

### プロファイルをどう取り、どう引き戻すか

PGO は「プロファイルを取って食わせるだけ」に聞こえるが、具体化すると難所が二つある。

一つ目は**プロファイルの取り方**。測定用のカウンタをコードに埋め込む**計装**方式は正確だが、
計測専用のビルドが要るうえ、多スレッドではカウンタの取り合いが計測そのものを歪める。これに対し
**サンプリング**方式（AutoFDO[](#cite:chen2016autofdo)）は、**最適化済みの本番バイナリ**を
ハードウェアの性能カウンタで時々覗くだけでよく、計装方式の 85% 程度の利得を取り戻す。

二つ目が**帰属（attribution）問題**——プロファイルを採ったバイナリと、それを使って最適化する
コンパイルとが食い違う、というズレだ。とくにインラインが効くと、一つのソース行が最適化後コードの
あちこちに散らばり、「どの計測値をどの命令に戻すか」が難しくなる。

> [!NOTE]
> **発展：サンプリングで採った回数の辻褄を合わせる。**
> サンプリングは「全部数える」のでなく「ときどき覗く」ので、測った回数には誤差が乗る。すると、ある分岐で
> 「入ってきた回数」と「出ていった回数の合計」がズレてしまう——本来どのブロックでも入＝出のはずなのに、だ。
> そこで、各ブロックの回数を一斉に微調整して、**測定値からのずれをできるだけ小さく**しつつ
> 「どのブロックでも入＝出」という制約を満たす割り当てを探す。これはグラフ上の流れの問題そのもので、
> **最小費用流**（ずれを「費用」とみなし、それが最小になる流れを求める）として効率よく解ける（LLVM の `profi`[](#cite:he2022profi)）。
> なお、サンプルした番地を最適化後のコードから元の命令へ正しく対応づける（帰属する）こと自体も難しく、
> その対策として CFG に直接目印を埋める疑似プローブ[](#cite:he2024csspgo)などが使われる。

### リンク後に並べ替える — ポストリンク最適化

コンパイラ内の PGO には原理的な限界がある——コンパイラは**最終バイナリのレイアウトを見られない**。
そこで、リンクが済んだバイナリを、プロファイルを使って直接並べ替え直す**ポストリンク最適化**が登場した。
代表が **BOLT**[](#cite:panchenko2019bolt)だ。最終バイナリ上で基本ブロックや関数を並べ替え、
**ホットなコードを近くに集めて命令キャッシュを効かせる**——FDO や LTO を掛けた上に、さらに最大 8% 速くする。
プロファイルを採ったバイナリと書き換えるバイナリがほぼ同一なので、先ほどの帰属問題がほぼ消えるのも効く。

> [!NOTE]
> **発展**：配置最適化の系譜。
> ホット/コールドを意識した基本ブロック配置は古典の Pettis–Hansen[](#cite:pettis1990positioning)に始まり、
> 命令キャッシュをより精密にモデル化した ext-TSP[](#cite:newell2020exttsp)へ洗練された。
> ポストリンク系には、再リンク方式でウェアハウス規模へスケールさせた Propeller[](#cite:shen2023propeller)、
> BOLT を高速・軽量化した Lightning BOLT[](#cite:panchenko2021lightningbolt)、要約だけで全 IR を読まずスケールする
> リンク時最適化 ThinLTO[](#cite:johnson2017thinlto)などがある。

## 部分評価 — プログラムを「専門化」する

### 二分法 — 既知の入力を先に計算してしまう

全プログラム最適化の最も理論的に美しい頂点が**[部分評価](#index:部分評価)**
（partial evaluation; 部分計算）だ[](#cite:futamura1999)[](#cite:jones1993)。
アイデアはこうだ——プログラム `p` の入力の**一部**が事前に分かっているなら、
その既知部分について**先に計算してしまった**専用プログラム `p'` を作れる。残りの入力だけを後で渡せば、
`p'` は `p` と同じ結果を、より速く返す。

```ruby
def power(base, exp)            # 一般の冪乗
  exp.zero? ? 1 : base * power(base, exp - 1)
end

# exp = 3 が既知だと分かっているなら、exp で部分評価して専用版を生成できる:
def power_3(base) = base * base * base   # ループも再帰も消えた専用コード
```

部分評価器（specializer）`mix` は、プログラムと「既知の入力」を受け取り、専用プログラムを返す。
`mix(power, exp: 3)` が `power_3` を生む、というわけだ。既知部分は計算し尽くし
（静的な計算）、未知部分は残す（動的な計算）——この「**二分法（binding-time analysis）**」が部分評価の核だ。

### フタムラ投影 — インタプリタが消える

部分評価が言語処理系にとって決定的に重要なのは、**[フタムラ投影](#index:フタムラ投影)**
（Futamura projection）[](#cite:futamura1999)があるからだ。インタプリタ `interp(program, input)` を、
「`program` は既知、`input` は未知」として部分評価すると——

```math
\mathit{mix}(\mathit{interp}, \mathit{program}) = \text{「その program 専用の実行コード」}
```

つまり、**インタプリタを特定プログラムで部分評価すると、そのプログラムのコンパイル結果が出てくる**。
インタプリタの「解釈のオーバーヘッド」（命令を一つずつ読んで分岐する部分）が、
既知の `program` について計算し尽くされて消えるからだ。インタプリタを書くだけで、
コンパイラが（部分評価器を通じて）手に入る——これがフタムラの 1971 年の洞察である[](#cite:futamura1999)。

この理論は長く美しい理屈にとどまっていたが、現代に実用化された。GraalVM の Truffle[](#cite:wurthinger2013)は、
言語インタプリタを書くだけで、**部分評価によって高性能 JIT コンパイラを自動生成**する基盤だ[](#cite:wurthinger2017)。
インタプリタのコードを実行時の型情報で部分評価し、解釈オーバーヘッドを溶かして専用機械語を吐く。
「One VM to Rule Them All」——一つの最適化基盤で、JavaScript も Ruby も Python も
高速に走らせるという野心は、フタムラ投影の現代的実装にほかならない。
（メタトレーシングという別アプローチで同じ目標に挑むのが PyPy[](#cite:bolz2009)で、
詳しくは[投機的最適化の章](speculative-optimization.md)で扱う。）

> [!NOTE]
> **発展**：Truffle が部分評価する相手は、観測した型に応じて自分を書き換える
> **自己最適化インタプリタ**[](#cite:wurthinger2012selfopt)だ。また「インタプリタ固有のオーバーヘッドだけを消し、
> プログラム本来の計算は残す」という理想（**Jones 最適性**）を厳密にどう達成するかは、なお研究が続いている[](#cite:brown2018jones)。

> [!TIP]
> 部分評価は「全プログラム最適化の極北」だ。脱仮想化・LTO・PGO が「全体を見て無駄を削る」のに対し、
> 部分評価は「既知の情報で、プログラムそのものを専用品に作り替える」。動的言語の JIT が
> 「実行時に観測した型でコードを特殊化する」のと、思想はまったく同じ——
> 違いは「いつ既知情報が手に入るか」だけだ。静的部分評価と動的特殊化は、一本の線でつながっている。

## まとめ

- 脱仮想化は CHA[](#cite:dean1995)/RTA[](#cite:bacon1996)で呼び先を一つに絞り、仮想呼び出しを直接呼び出し・インラインに変える。動的世界では投機的に行う。
- LTO は翻訳単位を IR のまま持ち越し、リンク時に全体を再最適化する。ThinLTO は要約でスケールさせる。
- PGO は実測プロファイルで「どこが熱いか」を最適化判断に与える。推測を実測で置き換える。
- 部分評価[](#cite:futamura1999)は既知入力でプログラムを専用化する。フタムラ投影により、インタプリタの部分評価がコンパイルになる——Truffle[](#cite:wurthinger2017)はこれを実用化した。

ここまでが静的（および静的に通用する）最適化の総まとめだ。
次の第 IV 部は、静的にはほとんど何も仮定できない**動的言語**の世界へ飛び込み、
「実行時の観測を武器に変える」最適化を正面から扱う。
